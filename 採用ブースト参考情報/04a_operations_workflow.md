## 業務サイクル全体像

### 運用の基本原則

> **サイクルは「目安」であり「制約」ではない。**
> 日次・週次・2週間・月次は標準的なリズムの目安であって、
> 運用担当はどの施策もいつでも実行できる。
> OSはすべての機能を常時利用可能な状態で提供し、
> 「今はこのサイクルだからこの機能は使えない」という制約は一切設けない。
>
> 例:
> - 週次チェック日でなくても、気になればいつでもKPIを確認できる
> - 2週間待たずに原稿修正を判断・実行できる
> - 月次レポートの途中経過をいつでもプレビューできる
> - アラートが出たら即座に対応できる（サイクルを待たない）

## 1. キックオフ・配信準備

### 1-1. 現状フロー

```
契約見込み〜契約完了
  ↓
① キックオフMTGの日程調整（契約確定前から開始可能）
   ※ 契約後だと配信開始が遅れるため、見込み段階で先行調整
  ↓
② キックオフMTG実施
   - 採用目標・ターゲットのヒアリング
   - 目標CPA・採用数の合意
   - レポート頻度・連絡手段の合意
   - 運用方針のすり合わせ
  ↓
③ 営業からの引き渡し情報 + キックオフで得た情報を統合
  ↓
④ ターゲット設計（WHO: 誰を採りたいか / WHAT: 何を訴求するか）
  ↓
⑤ 媒体アカウント開設（AirWork等）
  ↓
⑥ 求人原稿作成（1ポジション=1原稿 → 10〜15本に展開、40%ずつ内容変更）
  ↓
⑦ Indeed/求人ボックスでレビュー申請（Indeed: 5営業日）
  ↓
⑧ お客様に原稿確認依頼
  ↓
⑨ 承認後、配信開始 → CPC設定・予算設定
```

### 1-2. 現状の課題

| 課題 | 影響 |
|------|------|
| 引き渡し情報が口頭やLINEで断片的 | 運用開始時に情報の抜け漏れ |
| 原稿展開（10〜15本）が手作業 | 1社あたり数時間の初期コスト |
| レビュー待ち期間（5営業日）の管理が手動 | 見落とすと配信開始が遅延 |
| 配信開始日がスプレッドシートでバラバラに管理 | 「この会社いつから配信だっけ？」が頻発 |

### 1-3. OS導入後フロー

```
契約見込み〜契約完了（営業OS Phase 5〜6）
  ↓
① 顧客運用カード自動生成（営業フェーズの情報から先行作成可能）
   （企業情報・職種・エリア・予算・契約プラン・診断データを集約）
  ↓
② キックオフアジェンダ自動生成（業種・契約内容に応じたテンプレート）
   → 運用担当が確認 → お客様とキックオフMTG日程を調整
   ※ 契約確定前でも日程調整を先行開始できる
  ↓
③ キックオフMTG実施
   → MTG中の合意事項（目標CPA・採用数・ターゲット・運用方針）をOS上で記録
   → 顧客運用カードに自動反映
  ↓

  ↓
⑦ 配信開始 → 配信カレンダーに自動登録
```

## 2. データ確認・アラート対応（目安: 日次 / いつでも実行可能）

### 2-1. 現状フロー

```
出社
  ↓
① 「今日はどの会社をチェックすべきか？」を思い出す / スプレッドシートで確認
   → 各社の運用開始日がバラバラなため、把握が難しい
  ↓
② 該当する会社の媒体管理画面にそれぞれログイン
  ↓
③ 表示回数・クリック数・CPC・応募数を目視確認 or CSVダウンロード
  ↓
④ スプレッドシートに手入力
  ↓
⑤ 明らかな異常があれば対応（応募ゼロ、CPC急騰等）
```

**所要時間:** 1社15分 × 3〜4社/日 = 45分〜1時間

### 2-2. 現状の課題

| 課題 | 影響 |
|------|------|
| **どの会社を今日見るべきかわからない** | 見落とし・確認漏れのリスク |
| 各媒体に個別ログインが必要 | 12社×3媒体=最大36回のログイン |
| データ転記が手作業 | ミス・工数・データの鮮度低下 |
| 異常値に気づくのが遅い | 応募ゼロが3日続いてから気づく |
| 12社分のチェック状況が頭の中だけ | 属人的、引き継ぎ困難 |

### 2-3. OS導入後フロー

```
出社 → OSを開く
  ↓
① 「今日のタスク」画面に自動表示:
   - 今日チェックすべき会社（週次チェック日ベース）
   - アラートが出ている会社（赤バッジ）
   - 配信終了が近い会社（黄バッジ）
  ↓
② アラートがあればクリック → 詳細確認 → 対応
  ↓
③ アラートなし = 正常運用中 → 次の業務へ
```

**所要時間:** 5〜10分/日

## 3. KPI分析・異常値レビュー（目安: 週次 / いつでも実行可能）

### 3-1. 現状フロー

```
週次チェック日（会社ごとに異なる）
  ↓
① 各媒体の管理画面にログイン → データ取得
  ↓
② スプレッドシートに転記
  ↓
③ 求人ごとのCTR・CVR・CPC・CPAを確認
  ↓
④ パフォーマンスが悪い原稿を特定
  ↓
```

## 7. AI改善支援

### 7-1. 概要

運用担当の「改善のネタ出し・判断」をAIで支援する機能群。
12社×複数求人のPDCAを2人で回すために不可欠。

### 7-2. 機能詳細

#### 7-2a. 低パフォーマンス原稿の自動フラグ

| 項目 | 内容 |
|------|------|
| トリガー | 週次KPI集計時に自動実行 |
| フラグ条件 | CTR < 2.0% / CVR < 0.5% / CPA > 目標の150% |
| 表示 | 週次ヒートマップ上で赤表示 + 改善提案リンク |
| 運用担当の操作 | フラグを確認 → 改善提案を見る → 採用 or 自分で修正 |

#### 7-2b. AI改善提案生成（Claude API活用）

| 項目 | 内容 |
|------|------|
| 入力 | 現在の原稿 + KPIデータ + 業界ベンチマーク + 成功パターンDB |
| 出力 | タイトル改善案（3案）+ 本文改善ポイント + 改善理由 |
| 技術 | Claude API（プロンプト: 「以下の求人原稿のCTRを改善するタイトルを3案提案してください」） |
| 精度向上 | 過去の改善実績（Before/After）を学習データとして蓄積 → 提案精度が向上 |

#### 7-2c. A/Bテスト管理・有意差判定

| 項目 | 内容 |
|------|------|
| テスト設定 | 原稿A vs 原稿B を同時配信 |
| 計測期間 | 2週間（設定変更可） |
| 有意差判定 | χ²検定 or フィッシャーの正確確率検定（p < 0.05で有意） |
| サンプルサイズ警告 | クリック数が少ない場合「判定に必要なサンプル数に達していません」を表示 |
| 結果アクション | 有意差あり → 勝者を自動適用提案 / 有意差なし → 延長 or 新テスト提案 |

#### 7-2d. A/Bテストアイデア自動生成

| 項目 | 内容 |
|------|------|
| トリガー | 改善が必要な原稿が検出された時 |
| 生成例 | 「タイトルに日給を入れる vs 時給を入れる」「写真あり vs なし」「応募ボタン上部 vs 下部のみ」 |
| 根拠 | 業界のベストプラクティス + 自社の過去テスト結果 |

#### 7-2e. 顧客要望→改善ポイント自動抽出

| 項目 | 内容 |
|------|------|
| 入力 | 月次MTGの顧客発言タグ + メール/LINEのやり取り |
| 処理 | AIが「改善に繋がる要望」を自動抽出・分類 |
| 出力例 | 「面接に来ない」→ リマインド施策 or リクブースト / 「若い人が欲しい」→ 原稿の訴求変更 |
| 連携 | クロスセル検知エンジン（Phase 7-4）にシグナルとして送信 |
