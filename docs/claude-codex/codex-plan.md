# STP財務フロー Plan Review（全実装に向けた整理）

## 目的

STP（採用ブースト）側の財務運用を、プロジェクトスタッフが実運用できる状態まで仕上げる。  
Phase 1 で土台は整ったため、残課題を整理し、優先順位付きで全体実装計画に落とし込む。

## 現状（Phase 1 完了前提）

Phase 1 で完了している想定の主な改善:

- STPプロジェクトコンテキストの集約（`project-context.ts`）
- STPスコープのサーバー検証（取引作成/更新、状態操作、編集ページ直打ち対策）
- 取引フォームの按分UI改善（Checkbox → RadioGroup）
- 期日/予定日の2フィールド化（`paymentDueDate` + `scheduledPaymentDate`）
- 文言統一（`確認` → `確定`, `未確認` → `未確定`）
- `未確定に戻す` の追加（グループ制約付き）
- 候補画面の文言変更（`取引化`）
- STP按分テンプレート管理ページ + サーバー検証
- STPメニュー導線の改善

## 会話で合意済みの方針（仕様の前提）

- STPページで手動作成できる取引は、STPに関係する取引のみ
- 按分なし時は STP以外のプロジェクト/コストセンターは選べない
- 按分あり時は STPを含む按分テンプレートのみ選べる
- 按分テンプレート管理は STP側で「共通管理画面のSTPフィルタ付きエイリアス」として提供
- `確定` と `経理へ引き渡し` は分ける
- 確定後も、条件付きで `未確定に戻す` 可能にする
- 期日と予定日は分ける
  - 売上: `入金期日` / `入金予定日`
  - 経費: `支払期日` / `支払予定日`
- 候補画面の主アクション名は `取引化`
- 取引管理画面は、入力・編集・確定・再提出・引渡しの主画面
- 候補画面は「候補判定（取引化/保留/不要）」の画面へ発展させる

## 未実装・未完了の項目（会話ベース）

### A. 候補判定フロー（最重要・未実装）

- 候補の `保留` / `不要` 判定
- 判定結果の永続化（再検出しても毎回復活しない）
- `不要` の再表示ルール（ソース変更時の扱い）
- `保留` 候補の一覧/フィルタ
- 理由（reasonType + memo）の保存
- ソース変更時の `needsReview` / `要再確認` 運用

### B. 変動金額候補の本格対応（最重要・未実装）

- `amount=null` 候補の扱い改善
- 候補画面での簡易入力（推奨）
  - 金額
  - 税額/税率
  - メモ
  - 期日/予定日（必要最低限）
- または未入力なら `取引化不可` にする明確な制御
- 現状の `0円取引化` リスクの解消（`amount ?? 0` 系）

### C. 候補画面の高度な選択肢（未実装・拡張）

- `既存取引に統合`
- `翌月へ繰越`
- `分割して取引化`
- `ソース修正へ戻す` 導線

### D. システムプロジェクト管理の本実装（設計のみ、未実装）

- `SystemProjectBinding` テーブル導入
- `routeKey`（例: `stp`）と MasterProject/CostCenter/法人の正式紐付け
- 管理者のみ編集可能な管理UI（または当面UIなし）
- `project-context.ts` の内部実装をDB参照に差し替え

### E. プロジェクト管理全体の整理（未実装）

- プロジェクトテーブル管理
- プロジェクトごとのスタッフ権限管理（閲覧/編集）
- 経理プロジェクト（CostCenter）との紐付け管理
- 運営法人との紐付け管理（必要に応じて）

### F. 請求/支払グループ作成UX改善（未実装）

- 現状の「先に取引先選択」起点の改善
- 「未グループ化取引を複数選択してグループ化」起点
- 取引先×月のおすすめグループ案表示
- 複数取引先混在時のエラーUX改善

## 今後の実装フェーズ（全実装ロードマップ）

## Phase 2（候補判定の本体実装）※最優先

### 目的

候補画面を「検出して終わり」ではなく、「判定して運用する画面」にする。

### 実装項目

- `TransactionCandidateDecision`（仮）テーブル追加
- 候補ステータス:
  - `pending`
  - `converted`（取引化）
  - `held`（保留）
  - `dismissed`（不要）
- `candidateKey + targetMonth` のユニーク制約
- 理由保存（`reasonType`, `memo`）
- `needsReview` フラグ（ソース変更時）
- 候補一覧のフィルタ（保留/不要/要再確認）
- `取引化` / `保留` / `不要` のUI操作
- 候補判定結果を再検出に反映

### 受け入れ条件

- 一度 `不要` にした候補が毎回復活しない
- `保留` 候補を後で一覧から拾える
- `要再確認` が見える
- 判定履歴の意図が追える（理由付き）

## Phase 3（変動金額候補の本格対応）※最優先級

### 目的

`amount=null` 候補の0円取引化事故を防ぎ、STPスタッフが候補画面で最低限の入力を済ませられるようにする。

### 実装項目（推奨方針A）

- 候補画面で簡易入力UIを追加
  - 金額（必須）
  - 税額/税率（必要に応じて）
  - メモ
  - 期日/予定日の一部
- `overrideAmount` 等を候補判定テーブルに保存
- `取引化` 時に override 値を優先使用
- 入力不足の候補は `取引化` 不可にする
- エラーメッセージを明確化

### 受け入れ条件

- `amount=null` 候補を未入力のまま0円取引化できない
- 候補画面から必要最低限の入力ができる
- 取引化後に取引管理画面で詳細編集できる

## Phase 4（システムプロジェクト紐付けの本実装）

### 目的

`/stp` の文脈をコードだけでなく、安全に管理可能な設定構造にする。

### 実装項目

- `SystemProjectBinding` テーブル追加
  - `routeKey`（immutable）
  - `projectId`
  - `defaultCostCenterId`
  - `operatingCompanyId`（必要なら）
  - `isActive`
- `getSystemProjectContext()` の内部実装を `SystemProjectBinding` ベースへ差し替え
- `routeKey` はコード/シード管理（一般スタッフ編集不可）
- 管理者向け設定UI（別フェーズでも可）

### 受け入れ条件

- `/stp` の動作が `MasterProject.code` ベタ依存でなくなる
- STPの紐付け先を安全に管理できる
- 既存STP画面の挙動が維持される

## Phase 5（プロジェクト管理・権限管理の整理）

### 目的

将来の複数プロジェクト展開に耐える管理構造にする。

### 実装項目

- プロジェクト管理UIの責務整理
- スタッフ権限（閲覧/編集/会計操作）の整理
- 経理プロジェクト（CostCenter）紐付け管理UI
- 運営法人との紐付け管理
- 各プロジェクト配下から必要画面へアクセス可能な導線整備

### 受け入れ条件

- スタッフが自分のプロジェクト配下だけで必要操作に到達できる
- 権限境界が画面とサーバーで一致している
- プロジェクト追加時の運用がコード改修依存になりすぎない

## Phase 6（請求/支払グループUX改善）

### 目的

グループ作成を現場運用しやすいUXに改善する。

### 実装項目

- 未グループ化取引一覧から複数選択してグループ作成
- 取引先×月のおすすめグループ案
- 複数取引先混在時のエラー案内
- 取引先先選択方式は互換で残すか評価

### 受け入れ条件

- 現場が「どこから作るのか」で迷いにくい
- グループ作成の手数が減る
- 誤グループ化の事故が減る

## 横断タスク（各Phaseで必須）

- 型チェック（`npx tsc --noEmit`）
- 必要な migration の生成/適用
- STPスコープのサーバー検証（UIだけに依存しない）
- 経理側フローの回帰確認
- 変更点のドキュメント更新（STP向け操作フロー）

## 直近の実行優先順位（実務向け）

1. Phase 2（候補判定: 保留/不要/永続化）
2. Phase 3（変動金額候補の簡易入力 + 0円防止）
3. Phase 4（SystemProjectBinding 本実装）
4. Phase 6（請求/支払グループUX改善）
5. Phase 5（プロジェクト管理/権限管理の本格整理）

## メモ（重要）

- Phase 1 により「運用の土台」は改善済みだが、候補判定の本体はまだ未完成
- 特に `保留/不要` と `変動金額候補` を実装しないと、候補運用は現場で詰まりやすい
- `/stp` 文脈のサーバー強制は今後も最優先で維持する（UI制限のみは不可）
