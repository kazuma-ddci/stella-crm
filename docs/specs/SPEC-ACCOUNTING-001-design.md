      - 未処理件数（未仕訳の取引、未消込の入出金、按分未確定）
      - アラート:
        - 取引未申請（CRM契約があるのに取引レコードがない）

#### ⑨ AllocationTemplateOverride（按分テンプレート変更時の個別適用記録）

按分テンプレートの明細が変更された場合、影響する取引ごとに「変更前を維持 / 変更後を適用」を選択できる。
このテーブルは「変更前を維持」を選んだ取引に対して、旧テンプレート明細の按分率スナップショットを保持する。
レコードがない取引は最新テンプレートの按分率を使用する。

```prisma
model AllocationTemplateOverride {
  id                    Int       @id @default(autoincrement())
  transactionId         Int       // FK → Transaction
  allocationTemplateId  Int       // FK → AllocationTemplate
  snapshotRates         Json      // 変更前の按分率スナップショット [{"costCenterId": 1, "rate": 50}, ...]
  reason                String?   // 維持理由（任意）

  createdBy             Int       // FK → MasterStaff
  createdAt             DateTime  @default(now())

  // Relations
  transaction           Transaction        @relation(fields: [transactionId], references: [id])
  allocationTemplate    AllocationTemplate @relation(fields: [allocationTemplateId], references: [id])
  creator               MasterStaff        @relation("AllocOverrideCreator", fields: [createdBy], references: [id])

  @@unique([transactionId, allocationTemplateId]) // 同一取引×同一テンプレートの重複overrideを防止
}
```

**計算ルール**: PL計算時に取引の按分金額を求める際:
1. AllocationTemplateOverrideにレコードがあれば → snapshotRatesを使用
2. レコードがなければ → 最新のAllocationTemplateLine.allocationRateを使用

## 5. 業務フロー詳細

### 5.0 取引入力画面（プロジェクト側）詳細仕様

#### 入力項目
- type（売上/経費）
- counterpartyId（必須、選択式）
- expenseCategoryId（必須、選択式）
- amount（必須）
- taxAmount（自動計算、手修正可）
- periodFrom / periodTo（必須）
- allocationTemplateId or costCenterId（排他）
- contractId（CRM紐付け取引の場合）
- note
- attachments（複数アップロード）
- paymentDueDate（経費の場合の支払予定日、任意）
- paymentMethodId（経費の場合の決済手段、任意）
- isWithholdingTarget（源泉徴収対象、任意）
- withholdingTaxRate / withholdingTaxAmount / netPaymentAmount（源泉徴収ON時に自動計算・手修正可）
- memo / note は同義のため、UI表示名は「摘要・メモ」に統一する
- 取引先の入力は選択式のみ（自由入力不可）。候補にない場合はその場でCounterparty新規作成し、作成後に選択状態へ戻す

#### 表示条件
- 按分ON時: allocationTemplateId 必須、costCenterId は非表示
- 按分OFF時: costCenterId 必須、UIラベルは「プロジェクト」（「按分先」は使用しない）

#### プロジェクトページ作成ルール
- `/[project]/finance/*` で新規作成した取引は、按分OFFの場合 `costCenterId` を当該PJに自動設定
- ユーザーが変更を試みた場合は警告表示し、保存不可

#### CRM紐付け区分
- `contractId` が設定されたものを「CRM紐付け取引」として表示
- `contractId` 未設定を「非紐付け取引」として表示（例: AWS等）

#### 契約終了警告
- `contractId` 指定時に契約終了日 < 取引期間の場合、保存前ダイアログを表示
- 選択肢: `登録を続行` / `契約情報を確認` / `キャンセル`

### 5.1 売上フロー（プロジェクト側→経理側）

```
1. 取引候補の検出
   担当者が対象月を選択（1ヶ月ずつ処理）
   → CRM契約 + 定期取引 → 候補一覧表示
   → ソースデータ（契約金額等）が変更されている場合は「更新しますか？」と表示
   → 担当者が確認・選択 → 取引レコード生成（status: "unconfirmed"）

2. 取引の確認
   担当者が内容を確認 → status: "confirmed"

3. 請求グループの作成
   取引先を選択 → 確認済み取引をグループ化 → InvoiceGroup作成

4. 請求書PDF作成
   InvoiceGroup情報からPDF生成 → pdfPath保存
   → status: "pdf_created"

5. 請求書送付
   メール送信 or 送付記録 → InvoiceMail作成
   → status: "sent"

6. 経理へ引き渡し
   → status: "awaiting_accounting"

7. 経理が仕訳処理
   自動仕訳 or 手動仕訳 → JournalEntry作成
   Transaction.status: "journalized"

8. 入金確認
   BankTransaction登録 → 消込処理 → Reconciliation作成

9. 入金完了
   消込金額が請求金額に達したら → Transaction.status: "paid"
```

### 5.2 経費フロー（プロジェクト側→経理側）

```
1. 取引候補の検出（売上と同様）

2. 取引の確認 → status: "confirmed"

3. 支払グループの作成（必要な場合）
   取引先を選択 → 経費取引をグループ化 → PaymentGroup作成

4. 請求書発行依頼
   発行依頼テンプレートでメール送信 → InvoiceMail作成
   PaymentGroup.status: "requested"

5. 請求書受領
   ファイル名マッチングで自動紐づけ → 通知
   担当者確認 → PaymentGroup.status: "confirmed"

6. 経理へ引き渡し → Transaction.status: "awaiting_accounting"

7. 経理が仕訳処理 → JournalEntry作成

8. 支払い処理
   BankTransaction登録 → 消込処理

9. 支払完了 → Transaction.status: "paid"
```

### 5.3 消込フロー

```
1. 入出金を登録（BankTransaction）

2. 消込画面で紐づけ
   未消込の入出金を選択 → 対応する仕訳を選択 → 消込金額を入力

3. 金額一致の場合
   → Reconciliation作成 → 入金仕訳を自動生成 → 完了

4. 金額不一致の場合
   a) 一部入金: 消込額を入金額で登録。残額は未消込
   b) 振込手数料: 差額仕訳（支払手数料/売掛金）を自動生成
   c) 値引き: 差額仕訳（売上値引/売掛金）を自動生成
   d) 手動: 経理が差額仕訳を作成

   ※ b)〜d)で生成される差額仕訳はその場でプレビュー表示し、科目や金額を修正可能
   ※ 確定時に「このパターンをルールに追加しますか？」と提案する

5. 消込取り消し
   消込レコードを削除 → 入出金と仕訳が未消込に戻る
```

**決済手段ごとの仕訳回数の違い**:
| 決済手段 | 仕訳タイミング | 仕訳例 |
|---|---|---|
| 銀行口座 | 入出金時に1回 | 普通預金/売掛金、外注費/普通預金 |
| クレジットカード | 利用時＋引落時の2回 | ①通信費/未払金 ②未払金/普通預金 |
| 現金（立替） | 発生時＋精算時の2回 | ①旅費交通費/未払金 ②未払金/普通預金 |
| 仮想通貨 | 送金時に1回 | 外注費/仮想通貨（レート換算。為替差損益が発生する場合あり）|

自動仕訳ルール（AutoJournalRule）でこの分岐を設定する。クレカ・現金の2回目仕訳は、BankTransaction登録（引落/精算）時にAutoJournalRuleで自動生成する。

### 5.4 按分フロー

```
1. 按分テンプレートを事前登録
   プロジェクト間で合意 → テンプレート＋明細を保存
   明細の合計 = 100%（バリデーション）

2. 取引登録時に按分テンプレートを選択

3. 按分確定
   作成者のプロジェクト → 自動確定（AllocationConfirmation作成）
   他プロジェクト → 通知 → 確認・確定

4. 全プロジェクト確定 → 経理に渡る

5. PLの計算
   取引金額（税込） × 按分率 = プロジェクト別金額（動的計算）
   端数処理: 1円未満は切り捨て。端数（合計との差額）は最後の按分先に寄せる

6. テンプレート明細の変更
   → 影響する取引を全て表示
   → 一括変更 or 個別選択（変更前/変更後）
   → クローズ済み月が含まれる場合は経理管理者権限が必要
     （権限がなければテンプレート変更自体が不可）
```

### 5.4.1 按分テンプレート編集ポリシー

#### 変更種別ルール
- **按分率そのものの変更（配分方針変更）**: 新テンプレート作成（バージョン管理）。旧テンプレートは既存取引で引き続き参照される
- **未確定枠の割当確定等の明細変更**: 既存テンプレート編集可（影響確認画面を表示）

#### 影響確認
- 変更時、影響するTransaction一覧を表示
- 一括適用 or 個別に「変更前維持/変更後適用」を選択
- 「変更前維持」を選んだ取引にはAllocationTemplateOverrideレコードを作成

#### クローズ月関与時の制限
- **非管理者**: テンプレート編集自体不可（オープン月のみ適用という中途半端な変更は許可しない）
- **経理管理者**: 編集可、クローズ月含めて影響一覧を表示

### 5.5 自動仕訳フロー

```
1. 取引が経理に到着

2. 自動仕訳ルールをマッチング
   取引先 × 種別 × 費目で検索（priority昇順）
   最初にマッチしたルールを適用

3. マッチした場合
   → JournalEntry（status: "draft"）を自動生成
   → 経理が確認 → "confirmed"

4. マッチしなかった場合
   → 「要手動仕訳」として表示
   → 経理が手入力

5. 手動仕訳確定時
   → 「このパターンをルールに追加しますか？」提案
   → 追加する場合はルールをその場で修正可能
   → 既存ルールと競合する場合は優先順位設定を促す
```

### 5.6 自動仕訳ルール管理画面仕様

#### 優先順位
- `priority` 昇順で評価（小さいほど高優先）
- UIで並び替え（ドラッグ or 上下移動）可能
- 並び替え後は連番再採番して保存

#### 競合警告
- 同じ条件粒度（取引先/種別/費目の一致度が同等）のルール追加時に警告表示
- 保存は可能だが、priority設定を必須にする

#### 手動仕訳からルール追加
- 提案ダイアログで条件・借方/貸方・priorityを編集して保存可能

### 5.7 Counterparty 重複検知・統合フロー

#### 新規追加時重複チェック
- 入力名に対して類似候補を検索（前方一致/部分一致/正規化比較）
- 候補がある場合は確認ダイアログを表示し、`既存を選択` / `新規作成` を選ばせる

#### 定期重複チェック画面
- 重複候補ペア一覧を表示
- アクション: `統合する` / `別物として扱う`

#### 統合実行前の影響範囲確認
- 件数表示対象: Transaction, RecurringTransaction, BankTransaction, AutoJournalRule
- 重複ルール発生時は警告表示

#### 統合実行
- DBトランザクション内でFKを統合先へ付替え
- 統合元は `mergedIntoId`, `mergedAt`, `deletedAt` を設定
- ChangeLogに統合操作を記録

### 5.8 コメント・差し戻しフロー

#### 経理側アクション
- 差し戻し時に `commentType=return`、`returnReasonType` を必須入力
- Notification をプロジェクト担当者へ発行
- 対象Transaction.statusを `returned` に更新

#### プロジェクト側アクション
- コメント返信（添付可）
- `再提出` で Transaction.status を `resubmitted` に更新
- 再提出時に経理側へNotification発行

#### スレッド表示
- `parentId` によるスレッド表示
- 添付は Attachment.commentId で紐付け

### 5.9 通知ページ（/notifications）UI仕様

- 一覧表示対象: 自分宛 `recipientId = currentStaffId`
- フィルタ:
  - status（unread/read/in_progress/completed）
  - category（finance/accounting/kpi/system/other）
- 操作:
  - ステータス変更（statusChangedAt, statusChangedBy 更新）
  - linkUrl遷移
- ヘッダー:
  - ベルアイコンに未読件数バッジを表示

## 6. バリデーションルール

### 6.1 取引テーブル
- 取引先ID（counterpartyId）は必須
- 費目ID（expenseCategoryId）は必須
- 金額（amount）は0以上
- 発生期間From ≤ 発生期間To
- 按分テンプレートIDとコストセンターIDは排他的（どちらか一方のみ設定可能）
- 按分テンプレート選択時、コストセンターIDはnull
- 按分なしの場合、コストセンターIDは必須
- **発生期間ベースの月次按分**: periodFrom = periodTo（同日）の場合は全額をその月に計上。periodFromとperiodToが異なる月にまたがる場合は、日割りで各月に按分する（例: 2/22〜3/21 → 2月は7日分、3月は21日分）。PL表示時に動的計算する（取引レコード自体は分割しない）

### 6.2 仕訳明細
- 同一仕訳内のdebit合計金額 = credit合計金額

### 6.3 按分確定・テンプレート明細・テンプレートオーバーライド
- 同一取引×同一コストセンターの按分確定は1レコードのみ（DB一意制約で強制）
- 同一取引×同一テンプレートの按分オーバーライドは1レコードのみ（DB一意制約で強制）
- 同一テンプレート内の按分率合計 = 100%
- 未確定枠（costCenterId = null）を含められる
- 按分端数処理: 税込金額に按分率を掛け、1円未満は切り捨て。合計との差額（端数）は最後の按分先に寄せる

### 6.4 請求グループ
- 同一取引先の取引のみグループ化可能
- 税額はグループの小計から一括計算（明細ごとに計算しない）

### 6.5 消込
- 消込金額の合計が入出金の金額を超えないこと
- 消込金額の合計が仕訳の金額を超えないこと（一部入金は可）

### 6.6 月次クローズ
- クローズ・再オープンは**経理管理者のみ**実行可能（プロジェクト側にクローズ機能なし。閲覧のみ）
- クローズ済みの月の取引・仕訳・消込は編集不可
- 再オープンには理由が必須
- MonthlyCloseLog.projectId はプロジェクト別PLスナップショットの保持に使用する（プロジェクト側が独自にクローズ操作する機能ではない）

### 6.7 ポリモーフィック参照テーブルの排他制約

以下のテーブルは複数の参照先FKをnullableで持つが、アプリ層で「いずれか1つのみがnon-null」を保証する:

| テーブル | 排他FK | ルール |
|---|---|---|
| Attachment | transactionId, journalEntryId, bankTransactionId, commentId | いずれか1つのみnon-null |
| InvoiceMail | invoiceGroupId, paymentGroupId | いずれか1つのみnon-null |
| TransactionComment | transactionId, invoiceGroupId, paymentGroupId | いずれか1つのみnon-null（parentIdは別途スレッド用で排他対象外）|
| CryptoTransactionDetail | transactionId, bankTransactionId | いずれか1つのみnon-null |
| JournalEntry | invoiceGroupId, paymentGroupId, transactionId | 全てnull可（手動仕訳）。設定する場合はいずれか1つ |

Prismaではチェック制約を直接定義できないため、Server Action/APIのバリデーション関数で強制する。

### 6.8 InvoiceGroup 状態遷移

```
draft → pdf_created → sent → awaiting_accounting → partially_paid → paid
                                                  → returned（差し戻し）
         ↑
         └── pdf_created（明細変更時、PDFを無効化して戻る）

sent → corrected（訂正請求書を作成した場合）
```

**請求書番号の変更不可ルール**:
- invoiceNumberが採番された（pdf_created以降の）InvoiceGroupの番号は変更不可
- 運営法人の略称（abbreviation）を変更しても、既発行の請求書番号は変わらない（新規発行分から新略称を使用）
- 訂正請求書には新しい番号を採番する（元の番号は変更しない）

### 6.8.1 InvoiceGroup 操作制限詳細

#### 作成フロー
1. counterparty 選択
2. 当該取引先の `status=confirmed` かつ `invoiceGroupId IS NULL` の取引のみ表示
3. 請求情報入力（operatingCompanyId, bankAccountId, invoiceDate, paymentDueDate）

#### 状態別操作
- **draft**: 明細追加/削除、請求情報編集、PDF作成、削除可
- **pdf_created**: 明細追加/削除、請求情報編集可（PDFは無効化）
- **sent**: 直接編集不可、訂正請求書作成のみ
- **awaiting_accounting**: 編集不可、訂正請求書作成のみ
- **corrected**: 閲覧のみ

#### PDF無効化
- `pdf_created` 以降に明細/請求情報が変わった場合、PDFは無効扱い
- UIに「明細が変更されたためPDFが無効になりました。PDFを再作成してください。」を表示
- 実装は `pdfPath` を残してもよいが、画面表示上 `isPdfValid=false` 相当で管理する

#### 訂正請求
- `correctionType`: replacement（差し替え）/ additional（追加請求）
- 元請求書は `corrected` へ遷移
- 訂正請求書には新しい番号を採番する

#### 税額参照元
- PDF/送付メール/画面合計表示の正式税額は `InvoiceGroup.taxAmount` を使用
- `Transaction.taxAmount` は参考表示

#### 請求書作成+送信の統合導線
- InvoiceGroup詳細画面に2系統の操作を用意:
  - `PDF作成・保存`（単独）
  - `PDF作成して送付へ`（保存後、自動で送付画面へ遷移）
- 「ワンクリック」の意図は、PDF保存→送付画面遷移を連結したショートカット導線で満たす

### 6.9 PaymentGroup 状態遷移

```
before_request → requested → invoice_received → confirmed → paid
                                              → rejected → re_requested → invoice_received
```

### 6.9.1 PaymentGroup フロー詳細

#### 発行依頼
- `requestedPdfName` を必須入力
- `InvoiceTemplate.templateType=request` のみ選択可能

#### 受信PDF自動マッチング
- マッチングはまず完全一致で判定する
- 完全一致しない場合は、ファイル名正規化（拡張子大小文字・前後空白・全角半角の基本正規化）後に再判定する
- それでも一致しない場合は自動紐付けせず、スタッフ確認待ちの手動候補として通知する（誤紐付け防止を優先）
- 一致時にPaymentGroup候補へ紐付け、スタッフ確認待ち通知を発行
- スタッフが `確定` するまで状態は `invoice_received`

#### 確認画面
- 期待金額、PDF内金額、支払期限を確認
- 必要に応じて `expectedPaymentDate` を更新可能

#### 却下/再依頼
- 却下時は理由入力必須（コメント or 専用カラム）
- 状態遷移: invoice_received → rejected → re_requested → invoice_received
- 再依頼は `templateType=request` のテンプレートを使用

### 6.10 createdBy / updatedBy の例外テーブル

要望書 2.1.6「全データにスタッフID記録」の原則に対し、以下のテーブルはイベントログ的性質（作成後に更新しない）または専用の実行者カラムで代替するため、汎用の createdBy/updatedBy を設けない。

| テーブル | 代替カラム | 理由 |
|---|---|---|
| AllocationConfirmation | confirmedBy | 按分確定イベント。作成後の更新なし |
| AllocationTemplateOverride | createdBy（のみ） | スナップショット記録。updatedByなし |
| Reconciliation | performedBy | 消込イベント。作成後の更新なし |
| MonthlyCloseLog | performedBy | クローズ/再オープンイベント。作成後の更新なし |
| Attachment | uploadedBy | ファイルアップロード。メタデータ更新なし |
| InvoiceMailRecipient | （親の InvoiceMail 経由） | 送信先子レコード。親のcreatedBy/updatedBy/sentByで作成・送信者を辿れる。宛先自体の個別変更は親レコードの再作成で対応 |
| TransactionComment | createdBy（のみ） | コメントは追記型。投稿後の編集なし |
| Notification | senderId | システム自動生成。発信者はsenderIdで辿る |
| ChangeLog | changedBy | 変更履歴自体は不変のイベントログ |

上記以外の全テーブルには createdBy / updatedBy を設ける。

## 7. 既存ロジックの流用・移行

### 7.1 流用するファイル・関数

### 7.2 既存会計モジュールテーブルの廃止

| 既存テーブル | 新テーブル | 方針 |

### 7.3 既存変更履歴テーブルの扱い

### 8.1 PDF生成・プレビュー

- Noto Sans JPフォント（既存）を使用

- 保存先: `public/uploads/invoices/{year}/{month}/{fileName}`
- **PDFプレビュー機能**: 保存前にブラウザ上でPDFをプレビュー表示する。`@react-pdf/renderer`のBlobProviderまたはPDFViewerを使用し、確認→保存の2ステップにする

### 8.2 メール送信
- 既存のnodemailerを使用
- `src/lib/email/invoice-email.ts`に請求書メール送信関数を追加
- 運営法人メールアドレスのSMTP設定を使用（nullの場合はシステム共通設定）
- PDF添付対応

#### 8.2.1 送信UI状態遷移
- 送信実行後のUI:
  - 成功: トースト表示（送信完了）、InvoiceMail.status=sent、InvoiceGroup.statusを送付済みに更新
  - 失敗: エラートースト表示（エラー内容含む）、InvoiceMail.status=failed、`再送` ボタン表示
- 送信失敗時は InvoiceGroup.status を `sent` に更新しない（`pdf_created` または直前状態を維持）
- 手動送付記録（LINE/郵送等）を実行した場合のみ、InvoiceMail記録作成後に InvoiceGroup.status を `sent` に更新する
- 再送:
  - 既存InvoiceMailを更新して再送（新規InvoiceMailは作成しない）

### 8.3 ファイルアップロード
- 既存のアップロードAPIパターンを流用
- `src/app/api/accounting/attachments/upload/route.ts`を作成
- 対応形式: PDF, Word, Excel, 画像, CSV
- 最大10MB
- 証憑テーブルにメタデータを保存

### 8.4 変更履歴の記録
- Prismaミドルウェアで自動記録を検討
- または各Server Actionで明示的に記録
- 対象テーブル: 全テーブル（最低限: Transaction, JournalEntry, AllocationTemplateLine）

### 8.5 通知の発行
- `src/lib/notifications/create-notification.ts`にヘルパー関数を作成
- 各業務アクション（差し戻し、入金完了、按分確定依頼等）で呼び出す
- ヘッダーコンポーネントにベルアイコン＋未読件数バッジを追加

### 8.6 取引先の同期
- MasterStellaCompanyのCRUDアクションにフックを追加
- 新規作成時: Counterpartyテーブルにもレコードを自動作成（companyId紐づき）
- 更新時: Counterparty.nameも同期更新
- 削除時: Counterpartyは論理削除しない（過去取引データがあるため）

### 8.7 請求書番号の自動採番
- 既存のStpInvoiceNumberSequenceテーブルを拡張（operatingCompanyIdを追加）
- フォーマット: `{abbreviation}-INV-{YYYYMM}-{NNNN}`
- トランザクション内で採番（競合防止）
- 採番済み（pdf_created以降）の番号は変更不可。略称変更時も既存番号は維持
