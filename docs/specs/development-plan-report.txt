Stella CRM 開発計画書


■ 概要

Stella CRM における今後の開発予定項目を整理したものです。
大きく以下の6領域にわたる改修・新規開発を予定しています。

 1. 卸関連の再設計（名称変更・リスト統合・経理移行）
 2. 申請者管理／助成金（紐付け方式の変更・データ自動反映）
 3. 顧客進捗状況／貸金（管理単位の変更・フォーム連携）
 4. コンサル履歴の刷新（入力項目の再設計）
 5. BPO専用ページの新設（ベンダー向け・自社向けページ構築）
 6. 経理用シートの実装（権限制御付き経理専用画面）

1. 卸関連の再設計
━━━━━━━━━━━━━━━━━━━━

【1-1】文言変更・入力項目追加

＜変更内容＞
・「卸アカウント新規追加」→「顧客情報入力」に名称変更
・以下の入力項目を新規追加
  - 貸付利用有無
  - 助成金利用有無
  - 補助金対象額（税込）
  - 申請額

＜検討事項＞
新規フィールドの追加はDBスキーマの変更を伴います。既存データとの整合性（既存レコードへのデフォルト値設定やnull許容の判断）を考慮する必要があります。また「貸付利用有無」「助成金利用有無」は後述の貸金・助成金モジュールとの連携トリガーにもなり得るため、将来的なデータフローを見据えた設計が求められます。


【1-2】卸アカウントリストの廃止

＜変更内容＞
・現行の「卸アカウントリスト」を廃止
・「顧客リスト」への一元管理に移行

＜検討事項＞
既存の卸アカウントリストを参照している画面・ロジックの洗い出しが必要です。データ移行についても、既存の卸アカウントデータを顧客リスト側のデータ構造にマッピングする変換処理を設計する必要があります。参照整合性を壊さずに移行するため、段階的な切り替えを行う想定です。


【1-3】「卸アカウント管理」名称変更

＜変更内容＞
・「卸アカウント管理」→「顧客リスト」に名称変更

＜検討事項＞
UI上の表示名だけでなく、ナビゲーション・パンくずリスト・ページタイトル・権限設定のラベル等、システム全体で整合性のある変更が必要です。


【1-4】経理管理への移行

＜変更内容＞
・卸ページ内で管理している「ツール代」「請求／入金状況」等の項目を削除
・経理ページ（経理シート）側での一元管理に移行
・卸／コンサル／BPO すべての請求・入金管理を経理側へ集約し、経理担当が管理・請求対応を行う運用に変更

＜検討事項＞
単なる画面移動ではなく、データの所在と責務の移行を伴います。現在、卸ページ側で入力・表示されている請求・入金データを経理側のデータモデルへ統合する必要があり、既存データのマイグレーション戦略の策定が不可欠です。また、卸・コンサル・BPO という異なる事業形態の請求データを統一的に扱えるデータ構造の設計が重要です。事業形態ごとに請求サイクルや項目が異なる可能性があるため、柔軟性を持たせた設計が必要です。

2. 申請者管理（助成金）
━━━━━━━━━━━━━━━━━━━━

【2-1】紐付け方法の変更

＜変更内容＞
・現行の「LINE名での紐付け」から「顧客IDでの紐付け」に変更

＜検討事項＞
既存データの紐付け移行が最大の課題です。現在LINE名で紐付けられている申請者データを、正しい顧客IDに対応付ける必要があります。LINE名と顧客名の表記揺れ（全角/半角、敬称有無等）があるケースでは自動マッチングが困難な場合があり、手動確認が発生する可能性があります。また、LINE名が不明になることによる運用上の影響を確認し、必要に応じてLINE名を参考情報として残す設計も検討します。


【2-2】顧客リストからの情報自動反映

＜変更内容＞
・以下の情報を顧客リストから自動反映
  - 紹介元ベンダー

＜検討事項＞
顧客リストとのリレーション設計が必要です。顧客リスト側のデータが更新された場合の反映タイミング（リアルタイム参照 or バッチ同期）の判断が必要です。

3. 顧客進捗状況（貸金）
━━━━━━━━━━━━━━━━━━━━

【3-1】紐付け方法の変更

＜変更内容＞
・現行の「フォーム単位」での管理から「顧客ID単位」での管理に変更

＜検討事項＞
現在フォーム単位で管理されているデータを顧客IDに紐づけ直す必要があります。1つの顧客が複数のフォームを提出しているケースの扱い（最新のみ有効とするか、履歴として全件保持するか）の運用方針を決定する必要があります。


【3-2】借入申込フォームとの連携

＜変更内容＞
・借入申込フォームのリンクを、顧客IDごとに個別発行
・顧客リストの以下情報をフォームへ自動反映
  - 法人名 → 申請者名
  - 法人／個人区分
  - 補助金対象額（税込） → 借入希望金額

＜検討事項＞
顧客IDごとのフォームリンク発行は、URLにトークンやIDを含める方式が考えられますが、セキュリティ面での配慮が必要です（推測不可能なトークンの生成、有効期限の設定等）。また、フォーム送信時に顧客IDとの自動紐付けを行う仕組みが必要であり、外部フォームを使っている場合はWebhook等での連携設計が発生します。自動反映する項目について、顧客リスト側でデータが未入力の場合のフォールバック処理も考慮します。

4. コンサル履歴の刷新
━━━━━━━━━━━━━━━━━━━━

＜変更内容＞
現行のコンサル履歴の入力項目を以下に再設計します。

 1. タイトル（例：概要案内レクチャー）
 2. 議事録（Zoom等の議事録をそのまま貼り付け可能）
 3. 録画リンク（URL貼り付け）
 4. 先方タスク／弊社タスク（双方のアクションアイテムを管理）
 5. 次回アクション／次回期限（日付付きで次回予定を管理）
 6. 添付資料（ファイルアップロード対応）
 7. 備考（自由記述）

＜検討事項＞
議事録フィールドはZoomの自動生成議事録をそのまま貼り付ける想定のため、大量のテキストを扱える対応が望ましいです。リッチテキストエディタの導入はUI複雑度が増すため、まずはプレーンテキスト（改行保持）で対応し、将来的にMarkdown対応を検討する方針も選択肢となります。
「先方タスク／弊社タスク」は単一テキストフィールドとするか、チェックリスト形式で個別管理するかで実装工数が大きく変わるため、運用イメージを確認の上、設計を決定する必要があります。
添付資料のファイルアップロードは既存基盤を流用可能ですが、ファイルサイズ上限やストレージ容量の考慮が必要です。

5. BPO専用ページの新設
━━━━━━━━━━━━━━━━━━━━

【5-1】ベンダーページ

＜変更内容＞
・ベンダー向けの専用入力ページを新設
・入力項目の詳細は以下スプレッドシートに準拠
  https://docs.google.com/spreadsheets/d/1aD2kgPYr8byFCTC2i5FXSfzoNITk-KiC6Of6eRNua3o/edit?usp=sharing

＜検討事項＞
ベンダー（外部ユーザー）がアクセスする画面であるため、既存の外部ユーザー認証基盤を活用します。ベンダーごとに閲覧・編集できるデータのスコープ制御（自社分のみ表示）が必要であり、権限設計が重要です。また、ベンダーが入力したデータの妥当性チェック（バリデーション）も自社スタッフ向けより厳密に行う必要があります。


【5-2】自社側ページ

＜変更内容＞
・自社運用向けのBPO管理ページを新設
・入力項目：依頼日／事業者名／完了日
・フィルタ機能：完了日（対応日）での絞り込み（請求書発行用途）
・ステータス管理：予約済み／完了／不備／未完了
・表示順序：ベンダー名 → 事業者名 → 依頼日 → 対応日 → ステータス

＜検討事項＞
「完了日での絞り込み → 請求書発行」という運用フローを考えると、経理シートとの連携設計が重要です。BPOページで完了となったデータを経理側の請求データに反映する仕組み（自動 or 手動連携）を検討する必要があります。これは「1-4. 経理管理への移行」で述べた全事業統合の方針とも関連し、BPO・卸・コンサルの請求データを統一的に経理側へ流す仕組みの一部として設計するのが望ましいです。

6. 経理用シートの実装
━━━━━━━━━━━━━━━━━━━━

＜変更内容＞
・以下スプレッドシートの内容をシステム上で再現
  https://docs.google.com/spreadsheets/d/1mF1aQX_GRYDpkr8qS4vJLYOToFUjJVlbt7Nth0kuOS0/edit?usp=sharing
・閲覧・編集権限の制御：自社・ベンダー双方とも、経理担当者のみがアクセス可能とする

＜検討事項＞
権限制御が本機能の核心です。現行の権限モデルをベースに、「経理担当」という役割に対して適切なアクセス制御を設計する必要があります。特にベンダー（外部ユーザー）側の経理担当にのみ表示を許可する仕組みは、現行の外部ユーザー権限モデルの拡張が必要になる可能性があります。
また、スプレッドシートの内容をシステムに移行する際、スプレッドシート特有の柔軟性（自由なセル編集、関数による自動計算等）をどこまでシステム側で再現するかの線引きが重要です。業務上必要な機能に絞って構造化データとして設計する方針が現実的です。

領域間の依存関係
━━━━━━━━━━━━━━━━━━━━

各領域は独立しているように見えますが、以下の依存・関連があります。

▼ 顧客リスト（卸リスト統合後）
  → 申請者管理（助成金）：顧客IDで紐付け、紹介元ベンダー自動反映
  → 顧客進捗（貸金）：顧客IDで紐付け、フォームへの情報自動反映
  → BPOページ：事業者情報の参照

▼ 経理シート
  ← 卸ページ：請求・入金データの移行元
  ← コンサル：請求データの集約先
  ← BPOページ：完了データからの請求連携

「顧客リストの一元化」と「経理への請求集約」が全体のキーアーキテクチャであり、これらの基盤設計を先行して固めることが、各領域の開発をスムーズに進めるうえで重要です。

推奨開発順序
━━━━━━━━━━━━━━━━━━━━

Phase 1：卸関連の再設計（1-1〜1-3）
  → 顧客リスト一元化は他領域の前提基盤

Phase 2：経理用シートの実装
  → 請求集約の受け皿を先に構築

Phase 3：卸の経理移行（1-4）+ BPO専用ページ
  → 経理シート完成後に移行・連携

Phase 4：申請者管理 + 顧客進捗（貸金）
  → 顧客リスト基盤が整った後に着手

Phase 5：コンサル履歴の刷新
  → 独立性が高く、他領域への影響が少ない
