## 🤖 Claude（ターン 2）

これは経理システムを開発する上で非常によくある課題です。ドメイン知識がないと、何を聞けばいいかすらわからない「未知の未知」問題ですね。

## 進め方のアドバイス

### 1. まず自分の知識レベルを上げる（最低限のドメイン学習）

経理システムに必要な基本概念を先に学んでおくと、ヒアリングの質が劇的に変わります。最低限押さえるべきキーワード:

- **勘定科目** (売上、仕入、経費、資産など)
- **仕訳** (借方/貸方)
- **消費税** (税率、課税/非課税/免税の区分、インボイス制度)
- **請求書/領収書の法的要件** (インボイス番号、発行日、金額、税額など)
- **締め日・支払日サイクル**
- **売掛金/買掛金の管理**
- **月次決算/年次決算のフロー**

「経理 業務フロー 基本」「中小企業 経理 システム要件」などで検索して、2-3時間勉強するだけでもかなり違います。

### 2. ヒアリングの進め方を変える

知識がない状態で「何が必要ですか？」と聞いても、経理担当も「全部必要」としか言えません。代わりに:

**「今どうやってますか？」アプローチ**
- 「今の業務を1日の流れで教えてください」
- 「今使ってるExcelや帳票を全部見せてください」
- 「先月の月末に何をしたか、手順を教えてください」

これが一番効果的です。既存のExcelや紙の帳票には、必要な項目が全部入っています。

**「困ってることは？」アプローチ**
- 「今一番手間がかかってる作業は何ですか？」
- 「ミスが起きやすい部分はどこですか？」

### 3. 既存の帳票・Excelをもらう

**これが最も重要**。経理担当に「今使っているExcel、帳票、フォーマットを全部ください」とお願いしてください。それを見れば:

- どんな項目が必要か
- どんな計算をしているか
- どんな分類があるか

が一目でわかります。コードで言う「既存システムのリバースエンジニアリング」と同じです。

### 4. 既存の経理ソフトを参考にする

freee、マネーフォワード、弥生会計などのスクリーンショットやデモを見ると、「経理システムに必要な画面・項目」の全体像が掴めます。車輪の再発明を避けるためにも、既存製品の画面構成は非常に参考になります。

### 5. 段階的に作る

一気に完璧なものを目指さず:

1. **まず請求書の発行/管理だけ**作る
2. 経理担当に触ってもらってフィードバックをもらう
3. 「これに加えてXXも必要」という声を拾って拡張

小さく作って見せる → フィードバック → 改善、のサイクルを回すのが一番確実です。

---

要するに、**「聞く前に見せてもらう」**が鍵です。既存のExcelや帳票を手に入れれば、それがそのまま要件定義書になります。

## 🤖 Claude（ターン 4）

良い質問ですね。結論から言うと、**最初から同じプロジェクト（stella-crm）内で作る方がいい**です。

## 理由

### 同じプロジェクト内で作るべき理由

1. **データの連携が前提になっている**
   - 「どのプロジェクトのお金か」を紐づける = 既存の企業・契約・案件データと直接リレーションを張る必要がある
   - 別プロジェクトだとDB間の連携が非常に面倒になる

2. **stella-crmの構成がすでにマルチドメイン対応になっている**
   - 現状でも `/companies`（企業管理）、`/stp`（STP業務）とドメインが分かれている
   - `/finance` や `/accounting` を追加するのは自然な拡張

3. **認証・権限が共通**
   - ユーザー管理、ログイン、権限制御をそのまま使える
   - 経理担当には経理画面だけ見せる、といった権限制御も既存の仕組みで対応可能

4. **別プロジェクトで作ると「統合」が地獄になる**
   - DBスキーマの統合、認証の統合、UIの統合、デプロイの統合...
   - 「後で統合」は技術的負債の温床。やるなら最初から一緒の方が圧倒的に楽

### 具体的な進め方のイメージ

```
src/app/
  ├── companies/        # 既存: 企業管理
  ├── stp/              # 既存: STP業務
  ├── accounting/       # 新規: 経理機能
  │   ├── invoices/     #   請求書管理
  │   ├── payments/     #   入出金管理
  │   ├── reports/      #   経理レポート
  │   └── settings/     #   勘定科目など設定
  ...
```

Prismaスキーマも同様に、既存モデルとリレーションを張る形で拡張:

```
model Invoice {
  id          Int      @id @default(autoincrement())
  company     Company  @relation(...)  // 既存の企業と紐づく
  contract    StpContract? @relation(...)  // 既存の契約と紐づく
  ...
}
```

### 唯一「別にする」ケース

もし経理システムが完全に独立していて、stella-crmのデータと一切連携しないなら別プロジェクトもありですが、今回は「どのプロジェクトのお金か連携する」と明言されているので、分ける理由がありません。

---

なので、stella-crm内に `/accounting` 的なセクションを新設して、少しずつ機能を足していく形で進めましょう。まずは経理担当から既存のExcel・帳票をもらうところからですね。

## 🤖 Claude（ターン 6）

## 仕訳（しわけ）とは

簡単に言うと、**「お金の動きを、決まったルールで記録すること」**です。

### 日常で例えると

例えば「クライアントA社から50万円が振り込まれた」という出来事があったとき、ただ「50万円入った」ではなく:

```
【借方】普通預金  500,000円  ／ 【貸方】売掛金  500,000円
```

こう記録します。これが「仕訳」です。

### なぜこう書くの？

お金の動きには必ず**2つの側面**があるからです（複式簿記）:

- **何が増えた？** → 普通預金（銀行口座のお金が増えた）
- **何が減った？** → 売掛金（「もらう権利」がなくなった）

### よくあるパターンで理解する

| 出来事 | 借方（左） | 貸方（右） |
|---|---|---|
| A社に請求書を送った | 売掛金 50万 | 売上 50万 |
| A社から入金があった | 普通預金 50万 | 売掛金 50万 |
| 備品を現金で買った | 消耗品費 1万 | 現金 1万 |
| 電気代が口座から引落 | 水道光熱費 2万 | 普通預金 2万 |

### 借方・貸方のざっくりルール

- **借方（左）**: 資産が増えた、費用が発生した
- **貸方（右）**: 資産が減った、収益が発生した

左右は必ず同じ金額になります。これが「バランスする」ということ。

### 「勘定科目」との関係

「普通預金」「売掛金」「売上」「消耗品費」... これらが**勘定科目**です。仕訳で使うカテゴリ名のこと。会社ごとにある程度カスタマイズしますが、基本的な科目は決まっています。

---

### stella-crmとの関連で言うと

将来的に経理機能を作るなら、例えば:

- 契約が成立 → 「売掛金 / 売上」の仕訳が自動生成
- 入金確認 → 「普通預金 / 売掛金」の仕訳が自動生成

こういう **業務と仕訳の自動連携**ができると、経理担当がめちゃくちゃ楽になります。これが経理システムの価値です。

何か他にわからない用語とかあれば聞いてください。

## 🤖 Claude（ターン 8）

## 借方・貸方を超シンプルに説明する

正直、「借方」「貸方」という名前自体に意味はないです。**ただの「左」と「右」**だと思ってください。

歴史的な理由で変な名前がついてるだけで、名前から理解しようとすると混乱します。

### これだけ覚える

```
左（借方）  ／  右（貸方）
```

これだけです。

### じゃあ何を左右に書くの？

**「もらったもの」が左、「出ていったもの」が右**

例えばコンビニでコーヒーを買った場合:

```
もらったもの → コーヒー（費用）    ／  出ていったもの → 現金
    左                                    右
```

### もう少し例を

**① 給料をもらった**
```
左: 現金（もらった）  ／  右: 給料（労働を出した）
```

**② パソコンをカードで買った**
```
左: パソコン（もらった）  ／  右: 未払金（後で払う義務が出た）
```

**③ クライアントから振込があった**
```
左: 銀行預金（もらった）  ／  右: 売掛金（もらう権利が消えた）
```

### なぜこんなことをするの？

**お金の動きを「もれなく二重に記録する」ため**です。

- 現金が減った → 何に使った？
- 売上が立った → お金はどこにある？

片方だけだと追跡できない。両方書くから、後で「お金の流れ」が全部たどれる。これが複式簿記の本質です。

---

要するに **「左＝もらった側、右＝出た側」で、必ずセットで記録する**、それだけです。「借方・貸方」という言葉は忘れて「左・右」で考えた方が理解しやすいですよ。

## 仕訳をもう一度整理する

さっきの「左＝もらった、右＝出た」がわかれば、仕訳は簡単です。

### 仕訳 ＝ 「何が起きたか」を1行で記録すること

会社で何かお金が動くたびに、こう書く:

```
左: ○○○  100,000円  ／  右: ○○○  100,000円
```

**左右は必ず同じ金額**。これだけ。

### 実際の業務の流れで見てみる

stella-crmで管理してるような「企業との取引」を例にします。

---

**ステップ①：A社と契約して請求書を送った**

「お金をもらう権利」を手に入れた。まだ現金はもらってない。

```
左: 売掛金 500,000円  ／  右: 売上 500,000円
```
- 左（もらった）: お金をもらう権利（売掛金）
- 右（出した）: サービスを提供した（売上）

---

**ステップ②：A社から口座に振り込まれた**

権利が消えて、実際のお金になった。

```
左: 普通預金 500,000円  ／  右: 売掛金 500,000円
```
- 左（もらった）: 銀行口座にお金が入った
- 右（出た）: もらう権利が消えた

---

**ステップ③：外注費を払った**

```
左: 外注費 200,000円  ／  右: 普通預金 200,000円
```
- 左（もらった）: 外注さんの仕事（費用として記録）
- 右（出た）: 銀行口座からお金が減った

---

**ステップ④：消費税がある場合**

A社に税込550,000円で請求した場合:

```
左: 売掛金 550,000円  ／  右: 売上    500,000円
                            右: 仮受消費税 50,000円
```

左右の合計は両方550,000円でバランスする。こういう風に片側が2行以上になることもあります。

---

### まとめると

仕訳のやることは3つだけ:

1. **何が起きた？**（請求した、入金があった、経費を払った等）
2. **左に何を書く？**（もらったもの・増えたもの）
3. **右に何を書く？**（出たもの・減ったもの）

左右の「○○○」の部分に入るのが**勘定科目**（売掛金、普通預金、売上、外注費...）です。

---

### 経理システム的に言うと

経理担当がやってることは、日々この仕訳を1件ずつ入力する作業です。stella-crmに経理機能をつけるなら、この **「仕訳の入力・一覧・検索」が一番基本の画面**になります。

## 実際の経理の業務フロー

### 場面：A社との契約が決まった（税込55万円）

---

### ① 営業側（stella-crm側）がやること

「A社と契約決まりました、50万＋消費税5万です」と経理に伝える。今だと多分、Slackとかメールとか口頭で伝えてると思います。

---

### ② 経理がやること：請求書を発行する

まず請求書を作る。この時点で仕訳を入力します:

```
左: 売掛金     550,000円  ／  右: 売上       500,000円
                               右: 仮受消費税  50,000円
```

これは**経理が会計ソフトに手で入力している**のが一般的です。

入力画面のイメージとしては:

```
日付：     2026-02-22
摘要：     A社 ○○業務委託 2月分

借方科目    金額        貸方科目      金額
─────────────────────────────────────────
売掛金     550,000    売上        500,000
                      仮受消費税    50,000
```

「摘要」ってのは「何の取引か」のメモ書きです。

---

### ③ 実際にA社から振り込まれた時

銀行口座に55万円入った。経理は通帳やネットバンクで入金を確認して、また仕訳を入力:

```
日付：     2026-03-31
摘要：     A社 ○○業務委託 2月分 入金

借方科目    金額        貸方科目      金額
─────────────────────────────────────────
普通預金   550,000    売掛金      550,000
```

これで①の「売掛金（もらう権利）」が消えて、「普通預金（実際のお金）」に変わった。帳簿上つじつまが合う。

---

### ④ もし外注費を払う場合

この案件でB社に外注費20万＋消費税2万を払うなら:

```
日付：     2026-03-15
摘要：     B社 ○○案件 外注費

借方科目    金額        貸方科目      金額
─────────────────────────────────────────
外注費     200,000    普通預金    220,000
仮払消費税  20,000
```

---

### つまり経理の日常は

1. **何か取引が発生する**（契約、入金、支払い、経費など）
2. **仕訳を入力する**（日付、摘要、勘定科目、金額）
3. **月末にチェックする**（売掛金が残ってないか＝未入金がないか、など）

この繰り返しです。

---

### stella-crmに経理機能をつけるなら

一番価値があるのは**②の自動化**です:

- 営業が契約を登録 → 仕訳が自動で下書き生成
- 入金を登録 → 対応する売掛金の消込仕訳が自動生成

経理担当は内容を確認して「承認」するだけになる。これが手入力から解放されるポイントです。

## 🤖 Claude（ターン 14）

その理解で完璧です。

## 整理すると

毎回の仕訳で入力するのは:

1. **日付**
2. **摘要**（何の取引かのメモ）
3. **左の勘定科目と金額**（1行以上）
4. **右の勘定科目と金額**（1行以上）
5. **左の合計 ＝ 右の合計になっていること**

これが全取引に共通のフォーマットです。

---

### だから経理システムのDB設計で言うと

```
仕訳テーブル（JournalEntry）
  - id
  - 日付
  - 摘要
  - 作成者

仕訳明細テーブル（JournalEntryLine）
  - id
  - 仕訳id（FK）
  - 借方or貸方（"debit" or "credit"）
  - 勘定科目（FK）
  - 金額
```

1つの仕訳に対して、明細が複数行ぶら下がる形。そして左（debit）の合計と右（credit）の合計が一致しないと保存できないようにバリデーションをかける。

---

### つまり経理機能を作るなら最低限必要なのは

- **勘定科目マスタ**（売掛金、売上、普通預金、外注費... の一覧管理）
- **仕訳入力画面**（日付、摘要、左右の科目と金額を入力）
- **左右の合計一致チェック**
- **仕訳一覧・検索画面**

これが経理システムの一番核になる部分です。ここさえできれば、あとは「契約登録したら仕訳を自動生成」みたいな便利機能を足していく形になります。

## 業務フローのイメージ

```

        │                              │

## これをシステムに落とすと

### プロジェクト側が持つデータ

```

  - どの企業の
  - どの契約/案件の
  - 売上 or 経費
  - 金額、消費税
  - 請求書発行済みかどうか
  - ステータス（経理処理待ち → 入金済み）

```

  - どの取引に紐づくか（FK）

```
請求書発行 → 経理処理待ち → 仕訳済み → 入金済み（完了）
```

```

## 1つの取引に対して仕訳が複数発生する

さっきの例で言うと:

```
取引: A社 55万円の売上

  仕訳①（2/22）売上計上
    左: 売掛金 550,000 ／ 右: 売上 500,000 + 仮受消費税 50,000

  仕訳②（3/31）入金
    左: 普通預金 550,000 ／ 右: 売掛金 550,000
```

1つの取引に対して**タイミングの違う仕訳が2つ**ある。もし取引に直接明細をつけると、この2つの仕訳がごちゃ混ぜになる。

- どの明細が売上計上時のもので、どれが入金時のものか区別できない
- 仕訳日がそれぞれ違うのに、どの明細がどの日付か分からない
- 摘要も仕訳ごとに違う

---

## もっとややこしいケース

**分割入金の場合:**
```
取引: A社 55万円

  仕訳①（2/22）売上計上
  仕訳②（3/15）一部入金 30万
  仕訳③（3/31）残金入金 25万
```

3つの仕訳がある。それぞれ日付も金額も違う。これを仕訳テーブルなしで管理するのはかなりきつい。

---

## 要するに

```
取引テーブルに直接明細をつける場合:
  取引 → 明細（全部フラット。どれがいつの何の仕訳か不明）

仕訳テーブルを挟む場合:
  取引 → 仕訳①（2/22 売上計上）→ 明細
        → 仕訳②（3/15 一部入金）→ 明細
        → 仕訳③（3/31 残金入金）→ 明細
```

仕訳テーブルは**「いつ・何のために・まとめて記録した」というグループ**の役割。これがないと明細がバラバラになって、経理が月次で確認するときに追えなくなる。

だから6テーブル構成がいい。

## 理由

### 経理が実際にやることを考えると

経理は**請求書1枚に対して仕訳を1つ起こす**のが普通。

```
請求グループ #1（A社 2月分、税込165万）

  仕訳:
    左: 売掛金 1,650,000 ／ 右: 売上 1,500,000
                              右: 仮受消費税 150,000
```

初期費用と月額費用を別々に仕訳するわけじゃない。請求書1枚＝仕訳1つ。

### 入金も請求書単位で来る

A社は「初期費用分」「月額費用分」と分けて振り込まない。請求書の合計額で振り込んでくる。だから消込も請求グループ単位の方が自然。

```
入金: A社 1,650,000円
  → 請求グループ #1 の売掛金を消込
```

### もし取引単位にすると

```
取引①（初期費用）→ 仕訳A
取引②（月額費用）→ 仕訳B
```

入金が1回なのに仕訳が2つあって、どっちに紐づけるか面倒になる。経理が困る。

---

## ただし、按分は取引単位で必要

## まず按分先のプロジェクト問題

```

こうすればCRM外のプロジェクトも按分先にできる。

## 按分のフロー、どっちがいいか

### 案A：プロジェクト側が按分してから経理に渡す
```
取引: 家賃30万
  ↓ プロジェクト側が按分
  取引①: 家賃 PJ-A分 105,000円
  取引②: 家賃 PJ-B分  75,000円
  取引③: 家賃 PJ-C分 120,000円
  ↓ 3つの取引が経理に行く
  ↓ 経理がそれぞれ仕訳
```

**問題**: 取引レコードが膨大になる。家賃1つで3レコード、サブスク10個×3プロジェクトで30レコード。経理も面倒。

### 案B：経理が入出金後に按分する
```
取引: 家賃30万
  ↓ 経理に渡る
  ↓ 経理が仕訳＆入出金処理
  ↓ 処理完了後に按分を反映
```

**問題**: 経理が按分率を知らないし、経理の仕事が増える。

### 案C：取引は1つのまま、按分情報を付けるだけ ← おすすめ
```
取引: 家賃30万（レコードは1つ）
  └── 按分: PJ-A 35%
  └── 按分: PJ-B 25%
  └── 按分: PJ-C 40%

  ↓ 経理には取引1つとして渡る

```

**プロジェクト側と経理側の責任が完全に分かれる**:
- プロジェクト側 → 按分率を決める
- 経理側 → 仕訳と入出金をやる

## ベストな方法

**取引レコードは按分後の金額で作る。ただし按分元の情報を持たせて、按分率変更時は再生成する。**

```

- **プロジェクト側から見ると普通の取引**。「うちのPJに家賃105,000円かかってる」と見える
- **特別なフローじゃない**。按分後の取引も通常取引と同じステータス遷移

## 自動仕訳について

### 決済手段マスタ（PaymentMethod）
```

## FK = Foreign Key（外部キー）

テーブル同士を紐づけるためのカラムのこと。

### 具体例

```
仕訳テーブル
  id: 1
  請求グループID: 10  ← これがFK

請求グループテーブル
  id: 10
  企業ID: 5
  請求日: 2026/03/01
```

仕訳テーブルの「請求グループID: 10」が、請求グループテーブルの「id: 10」を参照している。この参照用のカラムがFK。

### なぜFKが必要か

「この仕訳はどの請求グループのものか」をデータとして持たないと紐づけられないから。FKがあることで:

```
仕訳#1 → 請求グループID: 10 → 請求グループ#10のデータが取れる
```

こうやって芋づる式にデータを辿れる。

### 今回のテーブルで使ってるFK一覧

```
取引.取引先ID            → 取引先マスタ.id
取引.按分テンプレートID    → 按分テンプレート.id
取引.コストセンターID      → 按分先マスタ.id
仕訳.請求グループID       → 請求グループ.id
仕訳明細.仕訳ID          → 仕訳.id
仕訳明細.勘定科目ID       → 勘定科目.id
消込.入出金ID            → 入出金.id
消込.仕訳ID              → 仕訳.id
```

全部「このレコードはあのテーブルのこのレコードに紐づいてますよ」という意味。Prismaだと `@relation` で定義するやつだね。

## 決済手段ごとの業務フロー

### 銀行口座の場合（一番シンプル）

```

② 入出金をシステムに登録
③ 仕訳と消込

```

### クレジットカードの場合

```
① カードの利用明細を確認（月1回まとめて届く）
② ただし引き落としは翌月

だから2段階の仕訳が発生する:

【利用時（3月）】
  取引: AWS 53,000円をカード払い
  仕訳: 通信費 53,000 / 未払金 53,000
  → まだお金は出てない。「後で払う義務」を記録

【引き落とし時（4月）】
  入出金: カード会社から引き落とし 53,000円
  仕訳: 未払金 53,000 / 普通預金 53,000
  → 実際にお金が出た。「後で払う義務」が消える
```

銀行口座と違って**2回仕訳する**のがポイント。

### 現金の場合

```
① 社員が立て替えて経費を使う
② 領収書を提出して経費申請
③ 経理が確認

【経費発生時】
  取引: タクシー代 3,000円
  仕訳: 旅費交通費 3,000 / 未払金 3,000
  → 社員に返す義務を記録

【社員に精算したとき】
  入出金: 社員に3,000円振込（or 現金手渡し）
  仕訳: 未払金 3,000 / 普通預金 3,000（or 現金 3,000）
```

現金もクレカと同じで**使った時と払った時で2回**。

### 仮想通貨の場合

これが一番複雑:

```

② 送金時のレートで日本円に換算する
③ レートが変動するから為替差損益が発生しうる

【送金時】
  取引: 接待交際費 $5,000（レート150.79 = ¥753,950）
  仕訳: 接待交際費 753,950 / 仮想通貨 753,950

  送金時: $5,000 × 150.79 = ¥753,950（時価）
  差額: 13,950円の為替差益

  仕訳:
    接待交際費   753,950 / 仮想通貨    740,000
                          為替差益      13,950
```

- まだ処理しないものまで出てくる

## 提案：自動生成の仕組みを変える

### 方針：「生成」じゃなくて「確認待ち一覧」にする

```
今の方式:
  ボタン押す → 3ヶ月分ドカッと生成 → 大量のレコードが並ぶ

新しい方式:
  システムが「来月発生するはずの取引」を自動検出
  → 確認待ち一覧として表示
  → 担当者が確認して「承認」したものだけレコード化
```

具体的には:

```
取引生成ダッシュボード - 2026年3月
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

対象月: [ 2026年3月 ▼ ]

■ CRMから自動検出された取引（未生成）
  ☑ A社 月額費用     500,000円  契約: A社保守  ← 契約から自動検出
  ☑ B社 初期費用   1,000,000円  契約: B社Web制作
  ☑ B社 月額費用     300,000円  契約: B社Web制作
  □ C社 成果報酬     500,000円  契約: C社採用  ← まだ確定してない

■ 定期取引から自動検出（未生成）
  ☑ オフィス家賃     330,000円  定期取引: 家賃
  ☑ AWS             50,000円  定期取引: AWS（変動・金額未入力）⚠️

■ 先月のソースデータ変更あり
  ⚠️ D社 月額費用 50万→60万に契約変更。反映しますか？

[選択したものを取引レコードとして生成]
```

### メリット

```
- 生成前に何が出てくるかプレビューできる
- 不要なものは外せる
- 金額が変わったものがわかる
- 月を選んで1ヶ月ずつ処理できる
- 変動金額のものは空欄で出てくるから後で入れられる
```

### ロジック的にやること

```
① CRMの契約データをスキャン
   → 対象月に有効な契約を抽出
   → 既に取引レコードがあるものは除外
   → 「未生成の取引候補」として表示

② 定期取引テーブルをスキャン
   → 対象月に該当する定期取引を抽出
   → 既に取引レコードがあるものは除外
   → 固定金額はそのまま、変動はブランクで表示

③ ソースデータ変更チェック
   → 契約の金額が変わってるのに取引レコードが古い金額のもの
   → 「更新しますか？」と表示
```

```
下書き:
  ✅ 取引の追加・削除
  ✅ 請求情報の編集
  ✅ PDF作成

  ✅ 取引の追加・削除（PDFは無効化される）
  ✅ 請求情報の編集（PDFは無効化される）
  ✅ PDF再作成
  ✅ 送付
  ✅ 削除

  ❌ 取引の追加・削除
  ❌ 請求情報の編集
  ✅ 訂正請求書の作成
  ✅ PDFの再ダウンロード
  ✅ 送付履歴の確認

経理処理待ち:
  ❌ 全て編集不可
  ✅ 訂正請求書の作成

  ❌ 全て編集不可（履歴として残る）

## ①消費税の端数処理と按分

### 明細にも消費税は表示する。ただし計算方法が違う

```
請求書の表示:
  明細①: 初期費用    1,000,000円
  明細②: 月額費用      500,000円
  ──────────────────────
  小計:             1,500,000円
  消費税（10%）:      150,000円  ← ここで一括計算
  合計:             1,650,000円
```

明細ごとに税額を出すんじゃなくて、小計に対して税額を1回だけ計算する。これがインボイス制度の正しいやり方。

### 按分の場合

```
取引: オフィス家賃 330,000円（税込）

按分テンプレート: PJ-A 40%、PJ-B 60%

PJ-AのPL表示:
  家賃(40%): 330,000 × 40% = 132,000円（税込で按分）

PJ-BのPL表示:
  家賃(60%): 330,000 × 60% = 198,000円（税込で按分）

合計: 132,000 + 198,000 = 330,000円 ✅
```

税込金額を按分するから端数問題が起きにくい。もし1円の端数が出た場合は、最後の按分先に寄せる（一般的なルール）:

```
100,001円を PJ-A 33%、PJ-B 33%、PJ-C 34% で按分:
  PJ-A: 33,000円
  PJ-B: 33,000円
  PJ-C: 34,001円  ← 端数をここに寄せる
  合計: 100,001円 ✅
```

## ⑦権限管理の説明

今のstella-crmにはスタッフの権限管理テーブルが既にある:

```
既存:
  MasterStaff（スタッフ）
  StaffPermission（権限）
  StaffRoleType（役割種別）
  StaffRoleAssignment（役割割当）
```

### やりたいことはシンプル

```
Aさん: STPの営業担当
  → STPの取引一覧は見れる・編集できる
  → 経理の画面は見れない
  → 他プロジェクトの取引は見れない

Bさん: 経理担当
  → 経理の画面は見れる・操作できる
  → プロジェクトの取引は見れるけど編集はできない

Cさん: 経理管理者
  → 経理の全操作ができる
  → 月次クローズ・再オープンができる
  → クローズ済み月の按分変更ができる

Dさん: 経営者
  → 全部見れる・全部操作できる
```

### 具体的にはページごとにアクセス制御する

```
/stp/finance/*         → STPプロジェクトの財務権限が必要
/pj-a/finance/*        → PJ-Aの財務権限が必要
/accounting/*          → 経理権限が必要
/accounting/monthly-close → 経理管理者権限が必要
```

### 権限の種類

```
プロジェクト側:
  finance:view      取引の閲覧
  finance:edit      取引の登録・編集
  finance:confirm   取引の確認（ステータス変更）
  finance:invoice   請求書の作成・送付

経理側:
  accounting:view     経理画面の閲覧
  accounting:edit     仕訳・消込の操作
  accounting:admin    月次クローズ、マスタ管理、
                      クローズ済み月の按分変更
```

### 実装イメージ

既存のStaffPermissionテーブルにこれらの権限を追加するだけ:

```
StaffPermission:
  staffId: 1（Aさん）  permission: "finance:edit"    projectId: STP
  staffId: 2（Bさん）  permission: "accounting:edit"  projectId: null（全社）
  staffId: 3（Cさん）  permission: "accounting:admin" projectId: null（全社）
```

ページを表示するときにチェック:

```
経理ページを開こうとした場合:
  Aさん → accounting権限なし → 「アクセス権限がありません」
  Bさん → accounting:edit あり → 表示OK
  Cさん → accounting:admin あり → 表示OK + 管理機能も表示
```

要するに「誰がどのページのどの機能を使えるか」を制御する仕組み。既存のテーブルに権限を追加するだけだから、新テーブルは不要。
