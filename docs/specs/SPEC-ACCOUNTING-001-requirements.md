# SPEC-ACCOUNTING-001: 経理・財務管理システム 要望書

## 1. 概要

stella-crmに全社横断の経理・財務管理機能を構築する。プロジェクト側（営業・PM）と経理側の業務を明確に分離し、それぞれが独立して稼働しつつ、データは連携する仕組みを実現する。

### 1.1 背景
- 現在STP（採用ブースト）専用の売上・経費機能が存在するが、これを全社横断の汎用的な仕組みに一新する
- 経理担当とのコミュニケーションを改善し、必要な項目・フローを網羅した設計を行う
- 今後プロジェクトが増えても全て同じ仕組みで管理できるようにする

### 1.2 対象範囲
- プロジェクト側：売上・経費の取引管理、請求書発行・送付
- 経理側：仕訳、入出金管理、消込、予実管理、キャッシュフロー予測
- 共通：取引先管理、通知、変更履歴

## 2. プロジェクト側の要望

### 2.1 取引レコードの管理

#### 2.1.1 取引の種別
- **売上**と**経費**の2種別を管理する
- CRMの契約データから取引レコードが自動生成される（各プロジェクトごとにロジックは異なり、その都度コーディングする）
- 定期取引テーブルからも取引レコードが自動生成される
- 手動での新規作成も可能

#### 2.1.2 取引の入力項目
プロジェクト側の業務担当者が入力する項目:
- **種別**（必須・選択）: 売上 / 経費
- **取引先**（必須・選択＋新規追加可）: 取引先マスタから選択。選択肢にないものはその場で新規追加できる
- **費目**（必須・選択）: 費目マスタから選択。売上用（初期費用、月額費用、スポット費用、その他等）と経費用（外注費、サブスク費、家賃、光熱費、交通費、広告宣伝費、消耗品費、その他等）に分かれる。マスタで管理し後から追加可能にする
- **金額**（必須・手入力）
- **消費税額**（必須・自動計算＋手修正可）: 金額入力時に10%で自動計算。軽減税率8%や非課税0円への修正も可能
- **発生期間**（必須・日付選択）: From〜To。初期費用は同日（例: 2026/2/22〜2026/2/22）で全額その月に。月額費用は期間指定（例: 2026/2/22〜2026/3/21）で按分される
- **按分**（任意・トグル）: ONにすると按分テンプレートを選択。OFFなら直接プロジェクトを指定
- **プロジェクト**（按分OFFの場合・必須・選択）: 按分先マスタ（CostCenter）から選択。按分OFFの場合のUI表示ラベルは「プロジェクト」とする（「按分先」とは表示しない）
- **按分テンプレート**（按分ONの場合・必須・選択）: 按分テンプレートマスタから選択
- **契約**（任意・選択）: CRMの契約に紐づく場合。選択した取引先の契約一覧から選択
- **源泉徴収**（任意・トグル）: ONにすると源泉徴収率（デフォルト10.21%）と源泉徴収額を自動計算。手修正可。差引支払額も自動算出される。経費で外注個人への支払い等に使用
- **支払予定日**（任意・日付）: 経費の場合いつ払うか
- **決済手段**（任意・選択）: どの口座/カードから払うか
- **摘要・メモ**（任意・手入力）
- **証憑**（任意・ファイルアップロード）: 複数ファイルアップロード可能。プロジェクト側からも経理側からもアップロードできる

#### 2.1.3 取引のステータス
```
未確認 → 確認済み → 経理処理待ち → 差し戻し → 再提出 → 経理処理待ち → 仕訳済み → 入金完了/支払完了
                                                                          ↓
                                                                        非表示（論理削除）
```

#### 2.1.4 プロジェクトページでの取引作成ルール
- どのプロジェクトページでも売上・経費を新規追加できる
- プロジェクトページで作成された取引は、按分なしを選んだ場合、自動でそのページのプロジェクトが選択される
- プロジェクトの選択を変更しようとした場合「ここは〇〇の作成場所です。他プロジェクトの取引は各プロジェクトページで作成してください」と表示する
- 接触履歴のプロジェクト・顧客種別のUI/UXと同じイメージ

#### 2.1.5 CRMデータとの整合性チェック
- CRMの契約データから「本来発生しているはずの取引」を自動検知し、未申請の場合はアラートを出す
- 取引登録時、紐づけた契約が終了している場合は警告を表示する:「この契約は〇月〇日に終了しています。取引を登録しますか？または契約情報を更新しますか？」
- 逆に、契約終了済みなのに取引が登録されている場合も経理ダッシュボードにアラート表示

#### 2.1.6 全データにスタッフID記録
- 原則、全てのテーブルに作成者（createdBy）・更新者（updatedBy）を記録する
- 重要なアクション（確認、承認、消込実行、メール送信等）には専用のカラム（confirmedBy、approvedBy、sentBy等）も設ける
- 後から「これを確定させたのは誰？」「メール送信したのは誰？」がわかるようにする（InvoiceMailにsentByカラムを設ける）

**例外（createdBy/updatedBy を持たないテーブル）:**

以下のテーブルはイベントログ的性質（作成後に更新しない）または専用の実行者カラムで代替するため、汎用の createdBy/updatedBy を設けない。

| テーブル | 代替カラム | 理由 |
|---|---|---|
| AllocationConfirmation | confirmedBy | 按分確定イベント。作成後の更新なし |
| AllocationTemplateOverride | createdBy（のみ） | スナップショット記録。作成後の更新なし |
| Reconciliation | performedBy | 消込イベント。作成後の更新なし |
| MonthlyCloseLog | performedBy | クローズ/再オープンイベント。作成後の更新なし |
| Attachment | uploadedBy | ファイルアップロード。メタデータ更新なし |
| InvoiceMailRecipient | （親の InvoiceMail 経由） | 送信先子レコード。親のcreatedBy/updatedBy/sentByで追跡 |
| TransactionComment | createdBy（のみ） | コメントは追記型。投稿後の編集なし |
| Notification | senderId | システム自動生成。発信者はsenderIdで辿る |
| ChangeLog | changedBy | 変更履歴自体は不変のイベントログ |

### 2.2 取引候補の検出・生成（一括生成の改善）

#### 2.2.1 現状の問題
- 「月次売上・経費を一括生成」ボタンで当月+2ヶ月分を一括生成する方式
- 何ヶ月分生成するか選べない
- まだ処理しないものが生成されても困る
- 全部手動も困る

#### 2.2.2 新しい方式：「確認待ち一覧」方式
- システムが対象月に発生するはずの取引を自動検出し、「取引候補」として一覧表示する
- 担当者が内容を確認し、チェックを入れたものだけ取引レコードとして生成する
- 対象月は自分で選択できる（1ヶ月ずつ処理）

#### 2.2.3 取引候補の検出元
- **CRMの契約データ**: 対象月に有効な契約から売上・経費を計算。既存のauto-generate.tsのロジックを活用
- **定期取引テーブル**: 対象月に該当する定期取引を抽出。固定金額はそのまま、変動金額は空欄で表示
- **ソースデータ変更チェック**: 契約の金額が変更されている場合、「更新しますか？」と表示

#### 2.2.4 各プロジェクトの生成ロジック
- 売上と経費の発生ロジックはプロジェクトごとに異なり、その都度指定・コーディングする
- 共通の取引レコードテーブルに格納されるため、生成後は全プロジェクト共通のフローに乗る

### 2.3 請求グループ（売上側の請求書管理）

#### 2.3.1 請求グループの作成フロー
1. 取引先を選択する
2. その取引先の「確認済み」かつ「未グループ化」の取引のみが表示される
3. グループ化する取引にチェックを入れる
4. 請求元法人・振込先口座・請求日・支払期限を設定する
5. 請求書番号は自動採番

#### 2.3.2 請求書番号のフォーマット
- `{法人略称}-INV-{年月}-{連番}`（例: STE-INV-202603-0001）
- 運営法人マスタに略称（abbreviation）カラムを追加し、そこで設定した値を使用する
- 略称を変更した場合、過去の請求書番号は変わらない（新規発行分から新略称を使用）
- 請求書PDFに記載されるため、発行済みの番号は変更不可

#### 2.3.3 請求グループの修正
- **取引の追加**: ステータスが「下書き」「PDF作成済み」の時のみ可能。同一取引先の確認済み・未グループ化の取引から追加
- **取引の削除（グループから外す）**: グループから外すだけで取引レコード自体は消えない。未グループ化に戻る
- **送付済みの訂正**: 直接編集不可。「訂正請求書を作成」ボタンで対応
  - 差し替え（新しい請求書で全体を置き換え）
  - 追加請求（差額分のみ追加で請求）
  - 元の請求書は「訂正済み」ステータスで履歴として残る

#### 2.3.4 ステータスごとの操作制限
- **下書き**: 取引の追加・削除、請求情報の編集、PDF作成、グループ削除が可能
- **PDF作成済み**: 取引の追加・削除、請求情報の編集が可能（PDFは無効化される。再作成が必要）。送付が可能
- **送付済み**: 全て編集不可。訂正請求書の作成、PDF再ダウンロード、送付履歴確認のみ
- **経理処理待ち**: 全て編集不可。訂正請求書の作成のみ
- **訂正済み**: 全て編集不可（履歴として残る）

#### 2.3.5 PDF作成済みで明細変更した場合
- PDFが古くなるため「明細が変更されたためPDFが無効になりました。PDFを再作成してください。」と表示
- PDF再作成ボタンを表示

### 2.4 支払グループ（経費側の請求書受領管理）

#### 2.4.1 支払グループが必要な理由
- 成果報酬等はそれぞれのタイミングで1件ずつ発生する。毎月まとめて支払うならそれらをまとめて相手に請求書発行を依頼する必要がある
- 請求書の発行依頼を自動で送れるようにしたい

#### 2.4.2 支払グループの作成フロー
1. 取引先を選択
2. その取引先の経費取引をまとめる
3. 支払予定日を設定
4. 相手に送付する請求書のPDF名を指定

#### 2.4.3 請求書発行依頼の自動送信
- 請求書発行依頼メールを自動で送れるようにする
- メールにはPDF名の指定を含める（例: 「代理店X_202603_Meta Trust宛.pdf」という名前で発行してください）
- 相手からメールでPDFが届いたら、ファイル名でマッチングし支払グループと自動紐づけする
- スタッフに通知が届き、「これであっていますか？」と確認画面を表示
- 確認して紐付きが正しければ「確定」を押す
- 支払予定日はこちら側で設定し、相手から届いた請求書の内容を見て必要なら更新する

#### 2.4.4 請求書の却下・再依頼
- 受領した請求書の内容に不備がある場合、「却下して再依頼」ボタンで対応
- 却下理由を入力し、取引先に再度発行依頼メールを送信できる
- 状態遷移: invoice_received → rejected → re_requested → invoice_received

#### 2.4.5 発行依頼用のテンプレート
- 請求書送付用テンプレートとは別に、請求書発行依頼用テンプレートも作成・管理できるようにする

### 2.5 請求書PDF作成・送付

#### 2.5.1 PDF作成
- 請求グループの情報からPDFを自動生成する
- 運営法人のロゴ画像を含める（運営法人マスタにロゴをアップロード）
- PDFプレビュー機能を作成する（ブラウザ上で表示）
- 保存すると請求グループレコードに紐づけて保存。すぐに閲覧・ダウンロード可能
- 既存の@react-pdf/renderer + Noto Sans JPフォントを活用

#### 2.5.2 メール送信
- 送信元メールアドレス: 運営法人メールアドレスマスタから選択
- 送信先: 取引先の担当者（StellaCompanyContact）から複数選択可能
- TO/CC/BCCを設定可能。手動でメールアドレスを入力することも可能
- テンプレートから件名・本文を自動生成。内容確認後に確定して送信
- 送信完了したら自動で「送付済み」ステータスに変更
- 送信成功: 「送信完了しました」をその場で表示
- 送信失敗: 「送信に失敗しました: エラー内容」をその場で表示。再送ボタンを表示

#### 2.5.3 メール以外の送付方法
- LINE、郵送、その他の送付方法にも対応
- 自動送付しなかった場合でも「送付を記録」で手動記録できる
- 送付方法（メール/LINE/郵送/その他）を記録するカラムを設ける

### 2.6 按分管理

#### 2.6.1 按分先マスタ（CostCenter）
- CRMのプロジェクトテーブルとは別に、経理専用のプロジェクトマスタを持つ
- CRMにあるプロジェクトはCRMプロジェクトIDで紐づけ、CRMにないプロジェクトは名称のみで管理
- これにより、CRM管理外のプロジェクト（社内開発、管理部門等）も按分先にできる

#### 2.6.2 按分テンプレート
- 事前にプロジェクト同士で按分割合を合意し、テンプレートとして登録する
- テンプレートには名称と明細（按分先＋按分率）を持つ
- 明細の合計は必ず100%でないと保存できない
- 按分先には「未確定」を選択可能。後でプロジェクトを割り当てるために残す枠
- +ボタンで入力行を追加していくUI

#### 2.6.3 按分テンプレートの変更ルール
- **按分率自体を変更する場合**: テンプレートのバージョン管理。新しいテンプレートを作成する（外注Xさんver2等）。過去の取引は旧テンプレートのまま
- **明細の変更（未確定→プロジェクト割り当て等）**: テンプレート自体を編集する。過去の分も変更される
- 明細変更時、影響のある取引を全て表示し、一括で全て変更するか、個別に変更前/変更後を選択できるようにする
- クローズ済みの月に関わる按分テンプレートの変更は、経理管理者権限が必要。権限がない場合はテンプレートの変更自体ができない（オープン月だけ変更という中途半端なことはしない）

#### 2.6.4 按分の計算
- 取引レコードは按分後の金額を直接持たない。テンプレートの按分率から動的に計算する
- PL表示時に「親取引の金額 × 按分率」で算出
- 端数処理: 税込金額を按分し、端数は最後の按分先に寄せる

#### 2.6.5 按分確定フロー
- 取引作成者のプロジェクトは自動確定
- 他プロジェクトには通知が届き、確認・確定してもらう
- 全プロジェクト確定後に経理に渡る

### 2.7 定期取引

#### 2.7.1 用途
- 毎月発生するサブスク費用、家賃等の定期的な経費を自動で取引レコードに生成するためのマスタ
- 売上にも対応するが、基本は経費用（売上はCRMの契約データから生成するため）

#### 2.7.2 定期取引の設定項目
- 種別（売上/経費）
- 取引先（取引先マスタから選択）
- 費目（費目マスタから選択）
- 名称（例: AWS利用料、オフィス家賃）
- 金額、消費税額、税率
- 金額タイプ（固定/変動）: 固定なら金額入りで生成、変動なら金額ブランクで生成
- 按分設定（按分テンプレート選択 or プロジェクト直接指定）
- 決済手段（決済手段マスタから選択、任意）
- プロジェクト（どのプロジェクトで使用するか、任意）
- 頻度（毎月/毎年/毎週）
- 実行日（毎月25日等）
- 開始日、終了日（null=継続中）
- 摘要・メモ
- 有効フラグ

### 2.8 取引先マスタ

#### 2.8.1 統一管理
- Stella全顧客マスタ（MasterStellaCompany）のデータは自動で取引先マスタにも同期する
- CRMにない取引先（AWS、Slack等のサービス）は取引先マスタで管理
- 企業IDで紐づくものと、名称のみのものが混在する

#### 2.8.2 重複防止
- 新規追加時に類似名称をチェックし「もしかして同じ取引先ですか？」と候補を表示
- 定期的な重複チェック画面を設ける
- 統合機能: 重複する取引先を1つに統合。過去の全取引データのFKを付け替える。統合元は論理削除。変更履歴に記録
- 統合前に影響範囲を確認画面で表示する（取引N件、定期取引N件、入出金N件等）

## 3. 経理側の要望

### 3.1 経理ダッシュボード
- 未処理件数（未仕訳の取引、未消込の入出金、按分未確定）を表示
- アラート（取引未申請、契約との矛盾等）を表示
- 今月のサマリー（売上、経費、入金、出金、未入金）を表示

### 3.2 仕訳管理

#### 3.2.1 仕訳の構造
- 仕訳テーブル（ヘッダー）: 仕訳日、摘要、作成者
- 仕訳明細テーブル: 借方or貸方、勘定科目、金額
- 1つの仕訳に複数の明細がぶら下がる。左（借方）の合計と右（貸方）の合計は必ず一致する
- 仕訳の紐づけ先:
  - **売上側**: 請求グループ（InvoiceGroup）単位で紐づく
  - **経費側**: 支払グループ（PaymentGroup）単位で紐づく
  - **個別取引**: グループ化されていない取引（Transaction）に直接紐づく場合もある
  - **手動仕訳**: どれにも紐づかない（経理が独自に作成する仕訳）

#### 3.2.2 仕訳の日付
- 仕訳日は経理側が管理する。プロジェクト側の発生日とは異なる場合がある
- 1つの取引に対して仕訳が複数発生する（売上計上時、入金時等、それぞれ日付が異なる）

#### 3.2.3 自動仕訳
- 取引が経理に来た時点で、自動仕訳ルールにマッチすれば仕訳の下書きを自動生成する
- 経理は内容を確認して「確定」するだけ。必要なら修正も可能
- マッチしなかった場合は「要手動仕訳」として表示し、経理が手入力する

#### 3.2.4 自動仕訳ルール
- 取引先 × 種別（売上/経費）× 費目の組み合わせで、借方・貸方の勘定科目を設定
- 優先度で並び替え可能。具体的なルールほど優先度を高く
- 同じルール（取引先と種別が同じ等、優先順位付けが必要なデータ）が存在する場合、追加時に警告を出し、優先順位の設定を促す
- 手動仕訳を確定した際に「このパターンをルールに追加しますか？」と提案する。追加する場合はその場で修正も可能

#### 3.2.x 将来連携（後続フェーズ）
- 将来的にSlackの稟議bot等から取引候補/申請データを自動取り込みできる拡張性を持たせる
- 本仕様では実装必須ではないが、取引のデータ元識別（例: manual / crm / recurring / slack）を持てる前提とする

#### 3.2.5 費目と勘定科目の関係
- 費目はプロジェクト側が選ぶ（外注費、サブスク費等のわかりやすい名前）
- 勘定科目は経理側で管理（外注費→科目コード7100等の正式名称）
- 費目→勘定科目の変換は自動仕訳ルールが行う
- 費目マスタにデフォルト勘定科目を持たせることで、自動仕訳ルールの補助にもなる
- プロジェクト側が費目を間違えた場合、経理側で修正可能

### 3.3 入出金管理

#### 3.3.1 入出金の登録
- 日付、区分（入金/出金）、決済手段、取引先、金額、摘要を登録
- 証憑（通帳コピー等）のアップロード可能（複数可）

#### 3.3.2 決済手段マスタ
- 種別: 現金、銀行口座、クレジットカード、仮想通貨ウォレット
- 名称（例: 三菱UFJ普通、会社用VISA、Trust USDT TRC）
- 種別ごとの詳細情報（JSON）: 銀行口座情報、カード情報、ウォレット情報等
- 初期残高、初期残高日（キャッシュフロー計算の起点）
- 残高アラート閾値（残高がこの金額を下回ったらアラート）
- クレカ用: 締め日、引落日、引落口座ID

#### 3.3.3 決済手段による仕訳の違い
- **銀行口座**: 入出金時に1回仕訳
- **クレジットカード**: 利用時（通信費/未払金）と引落時（未払金/普通預金）の2回仕訳
- **現金（立替）**: 発生時（旅費交通費/未払金）と精算時（未払金/普通預金）の2回仕訳
- **仮想通貨**: 送金時に仕訳。レート換算＋為替差損益の可能性あり

#### 3.3.4 仮想通貨取引
- 決済手段で仮想通貨ウォレットを選んだ場合、追加入力項目が表示される
- 銘柄（USDT、BTC、ETH等）、ネットワーク（TRC20、ERC20等）は固定選択肢で管理
- 初期リリースでは銘柄・ネットワークの選択肢は固定リストで運用する
- 運用上必要になった場合のみ、管理画面から追加可能な方式へ拡張する（初期段階では頻繁な追加を前提にしない）
- 取引先Walletアドレス、自社Walletアドレス
- 外貨金額、外貨単位（USD等）、レート、日本円金額（自動計算・修正可）

### 3.4 消込処理

#### 3.4.1 消込の概念
- 入出金（実際のお金の動き）と仕訳（会計上の記録）を紐づける作業
- 消込テーブルは多対多の中間テーブル

#### 3.4.2 消込パターン
- **入出金1つ→仕訳複数**: まとめ払い（例: 2ヶ月分一括入金）
- **入出金複数→仕訳1つ**: 分割払い（例: 一部入金＋残金入金）
- 消込テーブルに消込金額を持ち、振り分けを管理する

#### 3.4.3 金額が一致しない場合の処理
- **一部入金として処理**: 消込額を入金額で登録。残額は未消込。請求グループは「一部入金」ステータス
- **振込手数料として処理**: 差額分の仕訳（支払手数料/売掛金）を自動生成。売掛金が減って入金額と一致
- **値引きとして処理**: 差額分の仕訳（売上値引/売掛金）を自動生成。売掛金が減って入金額と一致
- **手動で仕訳を作成**: 経理が自分で差額の仕訳を作成
- 差額処理で生成される仕訳はその場でプレビュー表示され、科目や金額を修正可能
- 「このパターンをルールに追加しますか？」と提案される

#### 3.4.4 消込の取り消し
- 入出金管理画面から消込内容を表示し、個別に消込取消が可能
- 消込取消すると入出金と仕訳が未消込に戻る
- 消込テーブル自体の専用ページは不要（裏方テーブル）

### 3.5 コメント・差し戻し機能

#### 3.5.1 差し戻しフロー
1. 経理が取引を確認し問題があれば「差し戻す」
2. 種別（質問/修正依頼/承認確認/その他）とメッセージを入力
3. プロジェクト側に通知が届く
4. プロジェクト側がコメントに返信し、ファイル添付も可能で「再提出」
5. 経理側に再度届く

#### 3.5.2 コメント
- 取引または請求グループに対してコメントを付けられる
- コメント種別: 通常/差し戻し/承認/質問
- コメントのやり取りはスレッド形式で表示

### 3.6 勘定科目マスタ
- 科目コード、科目名、科目区分（資産/負債/収益/費用）
- 表示順、有効フラグ
- 経理管理者が管理

### 3.7 予実管理

- 本仕様のBudgetは経理/経営側の予算管理用であり、プロジェクト側の売上見込み（CRM契約/商談由来）とは別概念として扱う
- プロジェクト側の売上見込みは既存CRMデータから算出する想定とし、本仕様のBudgetテーブルとは分離する

#### 3.7.1 予算テーブル
- コストセンター別（全社 or プロジェクト別）× 勘定科目区分 × 年月で予算額を設定
- 年度の最初にまとめて入力。毎月同じなら一括コピー可能
- 定期取引から固定経費を自動で下書きに埋めるボタンを設ける
- 変動するもの（売上目標、外注費上限等）は手入力

#### 3.7.2 予実管理画面
- 予算 vs 仕訳実績を自動比較
- 差異、達成率を表示
- 差異が大きいところにアラート表示
- プロジェクト別にも表示可能

### 3.8 キャッシュフロー予測

#### 3.8.1 必要な情報
- **入金予定**: 請求グループの入金予定日（支払期限）から
- **出金予定**: 取引の支払予定日、定期取引の実行日、クレカの引落日＋利用額集計

#### 3.8.2 口座別残高予測
- 決済手段マスタの初期残高を起点に、入出金を足し引きして残高を計算
- 日別で残高推移を表示
- 残高アラート閾値を下回る見込みの場合、警告を表示

### 3.8.x 補足（初期フェーズの優先度）
- 予実/キャッシュフローのデータ構造と計算ロジックは初期設計に含める
- レポート画面の詳細な見た目・アウトプット設計は後続フェーズで段階対応とする

### 3.9 月次クローズ

#### 3.9.1 権限
- クローズ・再オープンは経理管理者のみ実行可能
- プロジェクト側にはクローズ機能なし（月次の数字を見るだけ）
- 権限がない人が操作しようとした場合「月次クローズ・再オープンは管理者のみ実行できます。管理者に依頼してください。」と表示

#### 3.9.2 クローズの効果
- クローズ済みの月の取引は編集不可
- 仕訳・消込も変更不可
- 再オープンで解除

#### 3.9.3 再オープン
- 理由の入力が必須
- クローズ・再オープンの履歴を全て記録（誰がいつ何をしたか）

#### 3.9.4 クローズ時のスナップショット
- 月次クローズ時にPLデータをJSONで保存する（MonthlyCloseLogテーブルにsnapshotDataカラム）
- クローズ済み月のPLはスナップショットの数値を使用

## 4. 請求・送信の要望

### 4.1 運営法人マスタ（既存）への拡張
- ロゴ画像パス（logoPath）: アップロードしたロゴが請求書PDFに使われる
- 略称（abbreviation）: 請求書番号の冒頭に使用（例: STE、MTR）

- メール件名テンプレート、メール本文テンプレートを設定
- テンプレート変数を使用可能（{{法人名}}、{{取引先名}}、{{担当者名}}、{{年月}}、{{合計金額}}、{{支払期限}}、{{指定PDF名}}等）

### 4.3 送信履歴
- 請求グループ・支払グループごとに送信履歴を記録
- 送信方法（メール/LINE/郵送/その他）を記録
- メールの場合: 送信元、送信先（TO/CC/BCC）、件名、本文を全て記録
- 送信ステータス（下書き/送信済み/送信失敗）
- 送信失敗時のエラーメッセージを記録

## 9. 消費税の端数処理ルール

### 9.1 請求書の税計算
- 明細ごとには税額を計算しない
- 請求グループの小計に対して一括で税額を計算する（インボイス制度準拠）
- 明細には税抜金額を表示し、合計欄で税額を一括表示

#### 9.1.x 税額の参照元優先ルール
- 取引レコードの税額カラムは入力/確認用の参考値として保持する
- 請求書PDFに記載する正式な税額は、必ず請求グループ（InvoiceGroup）の小計から一括計算した税額を使用する

### 9.2 按分の端数処理
- 税込金額を按分率で掛けて計算
- 端数（1円未満）が出た場合は最後の按分先に寄せる
