## 会話1

**ユーザー:**
契約書の送付をクラウドサインのAPIで自動化させたいと思ってます。
クラウドサインのAPIでどんなことができるか教えて。
やりたいことは、適切な相手に適切な内容が含まれた契約書を送付する。契約書の中身は金額など、個々で設定する。先方が締結したり破棄したりなどさまざまなアクションを読み取って、該当のレコードのステータスなどを変更する。また締結されたら書類を該当のレコードの証憑部分にアップロードするなどです。

**Claude:**
クラウドサインのAPIで実現できることを、やりたいことに対応させてまとめます。

### クラウドサイン Web API でできること

#### 1. 契約書の作成・送付

| エンドポイント | 用途 |
|---|---|
| `POST /documents` | 書類を新規作成（`template_id`でテンプレート指定可） |
| `PUT /documents/{id}/participants/{pid}` | 宛先（相手の名前・メールアドレス）を設定 |
| `POST /documents/{id}/files/{fid}/widgets` | 入力項目（フリーテキスト・チェックボックス・押印欄）を追加 |
| `PUT /documents/{id}/files/{fid}/widgets/{wid}` | 入力項目の内容を変更（金額等の個別設定） |
| `POST /documents/{id}` | 書類を送信 |

**やりたいこととの対応**: テンプレートを事前にクラウドサインに登録しておき、APIでそのテンプレートから書類を作成 → 宛先をセット → 金額などの入力項目を個別設定 → 送信、という流れが自動化できます。

#### 2. 先方のアクションを検知してステータス更新

**Webhook機能**で以下のイベントを検知できます:

| イベント | status値 | 意味 |
|---|---|---|
| `COMPLETED` | 2 | 締結完了 |
| `CANCELED` | 3 | 取り消し・却下 |
| `BOUNCED` | - | メール不達 |

Webhookは指定URLにPOSTで以下のJSONを送信します:
- `documentID` — 書類ID（CRMのレコードと紐づけるキー）
- `status` — 書類の状態
- `userID`, `email` — 送信者情報
- `text` — メッセージ本体

**やりたいこととの対応**: Webhookの受信エンドポイントをStella CRM側に作り、`documentID`で該当レコードを特定 → `status`に応じてレコードのステータスを更新、が実現できます。

#### 3. 締結済み書類のダウンロード・アップロード

| エンドポイント | 用途 |
|---|---|
| `GET /documents/{id}` | 書類の詳細データ取得 |
| `GET /documents/{id}/files/{fid}` | 締結済みPDFファイルのダウンロード |

**やりたいこととの対応**: Webhookで締結完了を検知 → APIで書類PDFをダウンロード → CRMの該当レコードの証憑としてアップロード、が実現できます。

#### その他の便利なエンドポイント

- `GET /documents` — 書類一覧取得（ステータス確認のポーリングにも使える）
- `PUT /documents/{id}` — タイトルやノートの変更
- `PUT /documents/{id}/decline` — 書類の取り消し
- `POST /documents/{id}`（送信済みの場合）— リマインド送信

#### 注意点

- アクセストークンの有効期限は **1時間**（定期的に再取得が必要）
- レートリミットは **1分間800リクエスト**
- API実行後のデータ反映に **数秒〜10秒** かかる場合がある
- Webhook通知先は最大 **20個** まで設定可能
- Webhookエラー時は **10分間隔で最大3回** 自動再送
- 対応プランは **スタンダード以上**

#### 実装イメージ

```
[Stella CRM] → POST /documents (テンプレートから書類作成)
            → PUT  /participants  (宛先設定)
            → PUT  /widgets       (金額等の入力項目設定)
            → POST /documents     (送信)

[CloudSign] → Webhook → [Stella CRM API]
                         ├─ COMPLETED → ステータス「締結済」に更新
                         │              → GET /files でPDFダウンロード
                         │              → 証憑としてレコードに保存
                         ├─ CANCELED  → ステータス「却下」に更新
                         └─ BOUNCED   → ステータス「送信エラー」に更新
```

やりたいことは全てAPIで実現可能です。実装に進める場合は、具体的な設計を詰めていきましょうか?

**Claude:**
はい、`documentID`はクラウドサイン側で自動生成されます。

`POST /documents`で書類を作成した際のレスポンスに`documentID`が含まれるので、その時点でCRM側のレコードに保存しておけば、後からWebhookで通知が来た時に紐づけられます。

流れとしては:

1. CRMから`POST /documents`で書類作成
2. レスポンスの`documentID`をCRMのレコードに保存
3. Webhookで`documentID`付きの通知が来る
4. 保存済みの`documentID`で該当レコードを特定 → ステータス更新

**ユーザー:**

**Claude:**
はい、できます。手動送信した書類でも`documentID`さえCRMに登録しておけば、Webhookで締結・却下を自動キャッチできます。

### documentIDの確認方法

クラウドサインの管理画面で書類を開いた際の**URLに含まれるID**が`documentID`です。例えば:

```
https://www.cloudsign.jp/documents/xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx
                                    ↑ これがdocumentID
```

もしくは`GET /documents`のAPIで書類一覧を取得して特定することもできます。

### 運用イメージ

```
【手動送信の場合】
1. クラウドサインのWeb画面から書類を手動送信
2. URLからdocumentIDをコピー
3. CRMの該当レコードにdocumentIDを貼り付け

【以降は自動】
4. Webhook通知 → documentIDで照合 → ステータス自動更新・PDF取得
```

API経由で送信した場合はステップ2-3が自動化されるだけで、Webhook側の処理は同じです。まずは手動送信+手動ID入力で始めて、後からAPI送信に移行するという段階的な導入もやりやすいと思います。

### 1. Webhook自動更新の制御（ケース3対応）

ケース3「API送付したがステータスは手動管理」を実現するには、Webhook更新を止めるフラグが必要。

```
cloudsignAutoSync: Boolean (デフォルト: true)
```

- `true` → Webhookで`cloudsignStatus`もCRM側`currentStatusId`も自動更新
- `false` → Webhookで`cloudsignStatus`は更新するが、CRM側`currentStatusId`は触らない

### 2. タイトルの同期ロジック

```
【API送付時】
  title → cloudsignTitle にコピー（初期値は同じ）

【Webhook受信時】
  if (title === cloudsignTitle)  → 両方更新
  if (title !== cloudsignTitle)  → cloudsignTitle だけ更新
```

ただし、クラウドサイン側からタイトルが変わることは基本的にない。

### 3. ステータスの同期ロジック（最重要）

CRM側はマスタで自由に定義できる（「社内確認中」「先方確認中」「締結済み」等）のに対し、クラウドサイン側は固定（draft / sent / completed / canceled）。

対応案: MasterContractStatus に `cloudsignStatusMapping` を追加

| CRMステータス | cloudsignMapping |
|---|---|
| 下書き | `draft` |
| 送付済み | `sent` |
| 締結済み | `completed` |
| 取消し | `canceled` |
| 社内確認中 | (なし) |

### 4. documentIDの再送付時の扱い（ケース5・7）

上書き方式。旧IDは`statusHistories`に記録。

### 5. テンプレート項目 → 契約日付のマッピング

`ContractType`レベルで設定:

```
ContractType:
  name: "業務委託契約"
  cloudsignTemplateId: "xxx-xxx"
  startDateWidgetLabel: "契約開始日"
  endDateWidgetLabel: "契約終了日"
```

### 6. 運営法人のAPIキー管理

- DBに平文で保存は避けるべき（暗号化 or 環境変数）
- 運営法人ごとにクラウドサインの契約が別 = トークン取得も別
- トークンは1時間で失効するので、取得・キャッシュ・リフレッシュの仕組みが必要

### 7. ファイル管理の注意点

締結前の書類を削除して良いか？提案: 両方残すか、ユーザーに選ばせる設定を設けるか。

### 8. 複数宛先への対応

クラウドサインは複数のparticipantを設定できる。送付フォームでの入力のみで、送付後は`cloudsignDocumentId`経由でAPIから取得する形で良い。
